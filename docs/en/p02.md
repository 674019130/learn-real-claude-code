# Agent Scaffolding: The Harness Matters

> The difference between a demo agent and a production agent is almost entirely in the scaffolding — the LLM call itself is the easy part.

## Overview

Every agent tutorial starts the same way: a while loop, an LLM call, a tool execution, repeat. In 20 lines of code you have a working agent. It makes API calls. It uses tools. It can complete tasks.

It also falls apart in production.

The gap between a demo agent and a production agent is not the quality of the prompts or the power of the underlying model. It is the scaffolding — the harness code wrapped around the LLM calls that handles everything the happy path ignores. Streaming. Concurrent tool execution. Context budgeting. Error recovery with circuit breakers. Tool scheduling. Output formatting that doesn't break under load.

Claude Code is 512,000 lines of code. The model interaction itself — the actual API call — is a few hundred of them. The other 511,000+ lines are the harness. This is not bloat. This is what production looks like.

## The Naive Agent vs. The Production Agent

The contrast between a minimal implementation and what Claude Code actually does is instructive:

```typescript
// Naive agent: works in demos, breaks in production
async function naiveAgent(task: string) {
  const messages = [{ role: 'user', content: task }];
  while (true) {
    const response = await anthropic.messages.create({
      model: 'claude-opus-4-5',
      messages,
      tools
    });
    if (response.stop_reason === 'end_turn') break;
    const results = executeTools(response.tool_use);
    messages.push(response, ...results);
  }
}

// Problems:
// - Waits for full response before doing anything
// - Runs tools sequentially even when they're independent
// - No retry on API failure
// - No context budget management
// - No circuit breaker — will retry forever on permanent failures
// - Entire context grows unbounded
```

```typescript
// Production agent: Claude Code's actual approach
async function productionAgent(task: string) {
  const stream = anthropic.messages.stream({ ... });  // Always stream

  for await (const event of stream) {
    if (event.type === 'content_block_delta') {
      renderDelta(event.delta);  // Show output immediately
    }
    if (event.type === 'tool_use') {
      scheduleToolExecution(event);  // Non-blocking scheduling
    }
  }

  // Execute independent tools concurrently
  const results = await executeConcurrently(
    pendingTools,
    { maxConcurrent: 5, respectSafetyDeclarations: true }
  );

  // Check context budget before next turn
  if (estimatedTokens() > CONTEXT_BUDGET_THRESHOLD) {
    await compressContext();
  }
}
```

The second version is more complex. It is also the one that works reliably at scale.

## The Five Pillars of Production Scaffolding

**1. Always Stream**

Waiting for a complete response before displaying anything is the wrong default. Users experience latency as silence — they don't know if the model is working or if something went wrong. Streaming shows work as it happens, which makes the same latency feel acceptable.

More importantly, streaming enables concurrent tool execution. Once you see a tool call in the stream, you can start executing it while the model continues generating. If the model generates three tool calls in a row, streaming lets you start all three before the model finishes generating the third.

**2. Declare and Respect Concurrency Safety**

Not all tools can run concurrently. Writing to the same file from two tools simultaneously corrupts the file. Reading the same file from two tools simultaneously is fine. Tools should declare their concurrency safety profile, and the scheduler should respect it.

```typescript
interface ToolDefinition {
  name: string;
  concurrencySafe: boolean;  // Can run with other tools simultaneously?
  writesTo?: string[];       // What resources does this tool write?
  readsFrom?: string[];      // What resources does this tool read?
}
```

A tool scheduler that understands these declarations can build a concurrency plan: run all read-only tools in parallel, sequence write tools that touch the same resource, isolate tools with unknown safety profiles.

**3. Budget Tokens Explicitly**

Infinite context is a fiction. Even with very large context windows, more tokens means higher latency and higher cost. Production agents should track context consumption and act on it — compressing history when approaching limits, prioritizing recent and relevant content over older content.

The budget is not just about hitting context limits. It is about maintaining performance. Context windows degrade as they fill (as discussed in the previous chapter). Explicit budgeting is the mechanism that prevents degradation.

**4. Circuit Break at the LLM Call Level**

Most developers think about circuit breakers at the tool execution level — if a tool fails repeatedly, stop calling it. This is necessary but not sufficient. The LLM call itself can fail — rate limits, service degradation, API errors. A circuit breaker at the LLM call level prevents the whole agent loop from spinning on a broken API endpoint.

The distinction matters because tool-level and LLM-level failures have different recovery semantics. A broken tool can often be skipped or worked around. A broken LLM call means the whole agent is down — that needs different handling.

**5. Format Output for the Consumer**

An agent that runs in a terminal needs different output formatting than one running in an IDE plugin or being called by another agent. Production scaffolding includes a formatting layer that adapts output to the consumer — markdown rendering for terminals that support it, plain text for consumers that don't, structured JSON for machine consumers.

## What Claude Code's Scaffolding Actually Does

Reviewing the Claude Code source through this lens, the scaffolding accounts for:

- Full streaming pipeline with delta rendering and concurrent tool scheduling
- Tool concurrency manager with read/write declaration and conflict detection
- Context budget tracker with automatic compression triggers
- 11-step exponential backoff + circuit breaker for API calls (covered in c14)
- Ink-based terminal UI with double-buffered rendering (covered in c12)
- Cost tracking integrated into the API call lifecycle
- MCP tool discovery and deferred loading (covered in c11)
- Multi-agent coordination via process spawning (covered in c10)

Every one of these is scaffolding. Not one is prompt engineering. The prompts are important, but they fit in a tiny fraction of the codebase. The harness is what makes those prompts work reliably in the real world.

## Key Takeaways

- The LLM call is a small fraction of a production agent; the scaffolding — streaming, concurrency, error recovery, context management — is the majority of the work
- Always stream: it improves perceived latency and enables concurrent tool execution during generation
- Tools should declare concurrency safety; the scheduler should enforce it to enable safe parallelism
- Explicit token budgeting prevents both context limit failures and the performance degradation that precedes them
- Circuit break at the LLM call level, not just the tool level — different failure modes require different recovery semantics
- Claude Code's 512K lines are mostly harness; the difference between demo and production is almost entirely found there
