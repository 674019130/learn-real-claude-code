# Context as a Resource

> Seven independent strategies fire in priority order: snip → microcompact → collapse → summarize — each targeting a different kind of context waste before escalating to the next.

## Overview

Context windows have limits. Every agent developer learns this eventually — usually the hard way, when a long session dies mid-task with a context overflow error. Claude Code treats context not as an infinite resource but as a budget that must be actively managed throughout the session's lifetime.

The compression system implements seven distinct strategies, each targeting a specific type of context waste. They fire in priority order: cheap and targeted strategies run first, more expensive and disruptive strategies are held in reserve. The goal is to maximize token recovery while minimizing information loss, with the model experiencing the compaction as seamless rather than jarring.

Understanding this system matters for two reasons: it explains why Claude Code can maintain coherent sessions over thousands of turns without user intervention, and the strategies themselves are a catalog of production-grade techniques you can apply in your own agents.

## When Compaction Triggers

The token budget module in `tokenBudget.ts` monitors context usage continuously in the agent loop. Compaction triggers at 95% of the context window:

```typescript
// tokenBudget.ts — trigger evaluation
function shouldCompact(usage: TokenUsage, model: ModelConfig): boolean {
  const contextWindowSize = model.contextWindow
  const currentUsage = usage.inputTokensUsed + usage.outputTokensUsed

  return currentUsage / contextWindowSize >= COMPACT_THRESHOLD  // 0.95
}
```

The 5% headroom is not arbitrary. Compaction itself requires tokens — the summarization request has to fit in the remaining space. A trigger at 99% would leave insufficient room for the compaction operation and risk failing at the worst possible moment.

Compaction fires *before* the next API call in the agent loop, so the model never experiences a truncated context mid-turn.

## The Seven Strategies in Priority Order

### Strategy 1: snip — Remove Duplicate System Messages

The cheapest strategy. Some operations inject system messages (from hooks, from `/commands`, from tool results). Over a long session, duplicates accumulate:

```typescript
// Before snip:
messages = [
  { role: 'system', content: 'You are working in /projects/myapp' },
  { role: 'user', content: '...' },
  { role: 'system', content: 'You are working in /projects/myapp' },  // duplicate
  { role: 'user', content: '...' },
  { role: 'system', content: 'MCP server connected: filesystem' },
  { role: 'system', content: 'MCP server connected: filesystem' },   // duplicate
]

// After snip:
messages = [
  { role: 'system', content: 'You are working in /projects/myapp' },
  { role: 'user', content: '...' },
  { role: 'user', content: '...' },
  { role: 'system', content: 'MCP server connected: filesystem' },
]
```

No information is lost. Tokens recovered can be significant in sessions that involve many tool calls, since each tool result may inject metadata.

### Strategy 2: microcompact — Summarize Recent Tool Results

Tool results are often verbose. A shell command that runs tests might produce 200 lines of output, of which only the final summary (pass/fail counts) is relevant after the immediate exchange. Microcompact targets tool result messages that are older than the last few turns:

```typescript
// microcompact: summarize old tool results
async function microcompact(messages: Message[]): Promise<Message[]> {
  const RECENCY_THRESHOLD = 6  // Keep last 3 exchanges (6 messages) verbatim

  return messages.map((msg, index) => {
    if (msg.role !== 'tool' || index >= messages.length - RECENCY_THRESHOLD) {
      return msg  // Keep recent messages and non-tool messages as-is
    }

    // Summarize old tool results to one line
    return {
      ...msg,
      content: summarizeToolResult(msg.content)
    }
  })
}
```

The recency threshold ensures the model always has full detail for recent context. Distant tool results that have already been reasoned about can safely be compressed to a one-line summary.

### Strategy 3: collapse — Collapse Long File Reads

File reads are often the biggest single consumers of context tokens. After the model has read a file and acted on its contents, keeping the full 1000-line file in context is wasteful:

```typescript
// collapse: replace file content with summary
async function collapse(messages: Message[]): Promise<Message[]> {
  return messages.map(msg => {
    if (!isFileReadResult(msg)) return msg
    if (isRecentMessage(msg)) return msg  // Don't collapse recent reads

    const filename = extractFilename(msg)
    const lineCount = countLines(msg.content)

    return {
      ...msg,
      content: `[File read: ${filename} (${lineCount} lines) — collapsed to save context]`
    }
  })
}
```

The model is informed that a file was read and its size. If it needs the content again, it can re-read it. This is preferable to silently truncating — the model knows what happened.

### Strategy 4: summarize — Full Conversation Summary

The most expensive strategy, held for last. This calls the model itself to summarize the conversation history, then replaces that history with the summary:

```typescript
// summarize: full conversation summary via model call
async function summarize(
  messages: Message[],
  context: AgentContext
): Promise<Message[]> {
  const summary = await callModel({
    system: SUMMARIZATION_SYSTEM_PROMPT,
    messages: [{
      role: 'user',
      content: `Summarize this conversation, preserving: goals, decisions made, ` +
               `current state, files modified, and any open questions.\n\n` +
               formatMessagesForSummary(messages)
    }]
  })

  return [
    {
      role: 'user',
      content: `[Conversation summary from auto-compaction:\n${summary}\n]`
    }
  ]
}
```

This strategy requires a full API call, which costs tokens and latency. It's a last resort — but it's better than losing the session entirely.

### Strategies 5-7

The remaining strategies handle edge cases:

- **prune**: Remove the oldest user/assistant exchanges entirely when the session is so long that even a summary would overflow
- **truncate**: Hard-truncate individual messages that exceed a per-message token ceiling
- **reset**: Nuclear option — retain only the system prompt and the last user message

Each escalates the disruption level. The system tries them in order and stops at the first strategy that brings context usage below a target threshold.

## The Execution Pipeline

```typescript
// Compaction pipeline in priority order
async function autoCompact(
  messages: Message[],
  context: AgentContext
): Promise<Message[]> {
  const strategies = [snip, microcompact, collapse, summarize, prune, truncate, reset]

  let current = messages
  const TARGET_USAGE = 0.70  // Aim to get below 70% after compaction

  for (const strategy of strategies) {
    current = await strategy(current, context)
    const usage = estimateTokens(current) / context.model.contextWindow

    if (usage < TARGET_USAGE) {
      return current  // Done — no need to escalate further
    }
  }

  return current  // Applied all strategies — return whatever we have
}
```

## Preserving Recency

All strategies share a common principle: recency is more valuable than history. The last few exchanges always receive the most lenient treatment. A message from 200 turns ago is a candidate for aggressive compression; a message from 2 turns ago is preserved verbatim.

This mirrors how human memory works in a working session. What you did an hour ago matters; what you did last week is background context at best.

## Key Takeaways

- Compaction triggers at 95% context usage — before the limit, not after — leaving headroom for the compaction operation itself.
- The seven strategies escalate in cost and disruption: cheap deduplication first, full summarization last.
- Each strategy targets a specific type of context waste: duplicates, verbose tool results, large file reads, full history.
- All strategies preserve recency: recent messages are always treated more conservatively than old ones.
- Truncation is always explained to the model — silent compression causes the model to act on information it no longer has, which is worse than knowing the content was removed.
