# Orchestrating 40+ Tools

> Tools declare isConcurrencySafe() — reads run in parallel, writes run exclusively. The scheduler knows what can race and what must wait.

## Overview

Forty-plus tools don't run themselves. There needs to be a registration system, a permission model, a concurrency scheduler, and a result management strategy — all working together so that the model can call any combination of tools in any order without corrupting state, blowing the context budget, or asking for permissions it already has.

`Tool.ts` defines the interface that every tool implements. `tools.ts` is the registry that collects and exposes them all. Together they form the tool orchestration layer that sits between the agent loop in `query.ts` and the actual implementations of filesystem access, shell execution, web search, and everything else.

This chapter explores the contract a tool must satisfy, how the concurrency scheduler uses `isConcurrencySafe()` to build an execution plan, how result budgets prevent context overflow, and why schema validation happens on both the way in and the way out.

## The Tool Interface — Tool.ts

Every tool in the system satisfies a single TypeScript interface:

```typescript
// Tool.ts (simplified)
interface Tool<TInput, TOutput> {
  // Identity & documentation
  name: string
  description: string

  // Input schema (Zod) — used for model-facing JSON schema AND runtime validation
  inputSchema: z.ZodType<TInput>

  // Core execution
  invoke(input: TInput, context: ToolContext): Promise<TOutput>

  // Result rendering (converts output to message content for the model)
  renderToolResult(output: TOutput): ToolResultContent

  // Permission & scheduling hints
  needsPermission(input: TInput): boolean
  isConcurrencySafe(input: TInput): boolean

  // Optional: result truncation budget (in tokens)
  resultBudgetTokens?: number
}
```

The Zod schema serves double duty: it's converted to JSON Schema for the model's tool definition (so the model knows what parameters to pass), and it's used at runtime to validate the model's actual input before `invoke()` is called. This means malformed tool calls — which do happen — are caught before they touch any real system state.

The `renderToolResult()` method deserves attention. Rather than dumping raw JSON back to the model, each tool controls how its output is formatted. A file read tool might return the raw content. A shell tool might format stdout/stderr/exit-code in a structured block. A web search tool might summarize results. The model receives whatever format is most useful, not whatever format is most convenient to produce.

## isConcurrencySafe() — The Concurrency Scheduler

When the model emits multiple tool calls in a single response, the `StreamingToolExecutor` has to decide: run them in sequence, or run them in parallel?

Running everything in sequence is safe but slow. Running everything in parallel is fast but potentially catastrophic — two write operations on the same file, or two shell commands that share state, can corrupt each other.

The solution is `isConcurrencySafe()`:

```typescript
// Examples from actual tool implementations

// ReadFileTool — pure read, always safe to parallelize
isConcurrencySafe(input: { path: string }): boolean {
  return true  // Multiple reads never interfere with each other
}

// WriteFileTool — mutates state, must run exclusively
isConcurrencySafe(input: { path: string, content: string }): boolean {
  return false  // Cannot run concurrently with any other write tool
}

// BashTool — depends on the command
isConcurrencySafe(input: { command: string }): boolean {
  // Heuristic: read-only commands (ls, cat, grep) are safe
  // State-mutating commands (rm, mv, git commit) are not
  return isReadOnlyCommand(input.command)
}
```

The executor builds an execution plan:

```typescript
// Simplified scheduling logic
async function executeToolCalls(toolCalls: ToolCall[]): Promise<ToolResult[]> {
  const safe = toolCalls.filter(tc => getToolFor(tc).isConcurrencySafe(tc.input))
  const exclusive = toolCalls.filter(tc => !getToolFor(tc).isConcurrencySafe(tc.input))

  // Run all safe tools concurrently
  const safeResults = await Promise.all(safe.map(executeOne))

  // Run exclusive tools sequentially
  const exclusiveResults = []
  for (const tc of exclusive) {
    exclusiveResults.push(await executeOne(tc))
  }

  return mergeInOriginalOrder(safeResults, exclusiveResults, toolCalls)
}
```

This is a simple but effective scheduler. The real implementation handles more edge cases (exclusive tools that might conflict with each other, tools that have data dependencies), but this captures the core idea.

## Result Budgets — Preventing Context Overflow

A naive tool implementation returns whatever it produces. A file read tool returns the whole file. A shell tool returns all stdout. This is fine for small results — catastrophic for large ones.

Consider: reading a 500KB source file into the context window eats most of the available tokens for a typical conversation. If the model then asks to read three more files, you're over the limit before any actual work happens.

The solution is `resultBudgetTokens`:

```typescript
// ReadFileTool — with budget enforcement
resultBudgetTokens = 20_000  // ~15KB of text

renderToolResult(output: FileContent): ToolResultContent {
  const tokenCount = estimateTokens(output.content)

  if (tokenCount <= this.resultBudgetTokens) {
    return { type: 'text', text: output.content }
  }

  // Truncate with explanation
  const truncated = truncateToTokenBudget(output.content, this.resultBudgetTokens)
  return {
    type: 'text',
    text: `${truncated}\n\n[File truncated: ${tokenCount} tokens, showing first ${this.resultBudgetTokens}]`
  }
}
```

The truncation isn't silent — it includes a message telling the model what happened. This matters: if the model doesn't know the result was truncated, it might act on incomplete information. With the explanation, it can decide whether to request a specific portion of the file instead.

## Schema Validation — In and Out

Most agent frameworks validate tool inputs. Fewer validate tool outputs. Claude Code does both:

```typescript
// Invoke with validated input
async function executeOne(toolCall: ToolCall): Promise<ToolResult> {
  const tool = registry.get(toolCall.name)
  if (!tool) throw new UnknownToolError(toolCall.name)

  // Validate input against Zod schema
  const parseResult = tool.inputSchema.safeParse(toolCall.input)
  if (!parseResult.success) {
    return formatValidationError(parseResult.error)
  }

  // Execute
  const rawOutput = await tool.invoke(parseResult.data, context)

  // Validate output (if outputSchema defined)
  if (tool.outputSchema) {
    const outputParseResult = tool.outputSchema.safeParse(rawOutput)
    if (!outputParseResult.success) {
      // Tool returned something unexpected — log and return error
      return formatOutputValidationError(outputParseResult.error)
    }
  }

  return tool.renderToolResult(rawOutput)
}
```

Output validation catches bugs in tool implementations early. If a tool is supposed to return `{ path: string, content: string }` but returns `{ filename: string, data: Buffer }`, the schema catches it before it reaches `renderToolResult()` and produces garbled output.

## Permission Model — needsPermission()

Not all tools require user confirmation. Reading files doesn't need a confirmation dialog. Deleting files should. The permission model is declarative:

```typescript
// DeleteFileTool
needsPermission(input: { path: string }): boolean {
  return true  // Always confirm before deletion
}

// GlobTool (file pattern matching)
needsPermission(input: { pattern: string }): boolean {
  return false  // Read-only enumeration, no confirmation needed
}

// BashTool — dynamic based on command
needsPermission(input: { command: string }): boolean {
  return !isWhitelisted(input.command)
}
```

Before `invoke()` is called on a permission-requiring tool, the executor pauses and requests confirmation. This integrates with the 5-layer permission stack covered in Chapter 8 — `needsPermission()` is layer 3 of that stack.

## Key Takeaways

- Every tool implements the same `Tool<TInput, TOutput>` interface — name, schema, invoke, renderToolResult, needsPermission, isConcurrencySafe.
- `isConcurrencySafe()` allows read operations to run in parallel while write operations run exclusively — this is a scheduling API, not just documentation.
- Result budgets (`resultBudgetTokens`) prevent large tool outputs from consuming the entire context window, and truncation is always explained to the model.
- Schema validation with Zod happens on both tool input and (optionally) tool output — bugs in tool implementations are caught before they produce confusing model behavior.
- `getTools()` in `tools.ts` is the central registry — adding a new tool means implementing the interface and registering it there.
