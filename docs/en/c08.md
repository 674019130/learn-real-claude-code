# Defense in Depth

> Hooks and the YOLO classifier race concurrently with user confirmation — the first to return a safe verdict wins, minimizing latency without sacrificing protection.

## Overview

An agent that can execute shell commands, write files, and make network requests has the capability to cause serious damage. The question isn't whether to restrict it — it's how to build restrictions that are robust without being so cautious they become useless.

Claude Code's answer is a five-layer permission stack. Each layer targets a different failure mode: the innermost layer prevents the most catastrophic operations unconditionally, and each outer layer adds more nuanced, configurable control. Crucially, the layers aren't sequential checkpoints — several operate concurrently, and the architecture races them against each other to minimize the latency cost of safety.

This chapter maps the full permission stack, examines how each layer works, and explains the racing mechanism that makes the system feel fast despite multiple concurrent safety checks.

## The Five-Layer Stack

```
┌─────────────────────────────────────────────────────┐
│  Layer 1: Hardcoded Safety                          │
│  "Never, regardless of any other config"            │
├─────────────────────────────────────────────────────┤
│  Layer 2: Permission Configuration                  │
│  allowedCommands, allowedPaths, blockedCommands     │
├─────────────────────────────────────────────────────┤
│  Layer 3: Tool-Level Permission Check               │
│  needsPermission() on the Tool interface            │
├─────────────────────────────────────────────────────┤
│  Layer 4: User Confirmation Dialog                  │
│  Interactive Y/N prompt for risky operations        │
├─────────────────────────────────────────────────────┤
│  Layer 5: YOLO Classifier                           │
│  AI classifier that assesses operation safety       │
└─────────────────────────────────────────────────────┘
```

### Layer 1 — Hardcoded Safety

Certain operations are blocked unconditionally in the source code. No configuration can override them, no user confirmation dialog enables them. Examples:

```typescript
// Hardcoded in tool implementations — checked before anything else
const HARDCODED_BLOCKS = [
  /^rm\s+-rf\s+\/(\s|$)/,          // rm -rf / and variants
  /^chmod\s+-R\s+777\s+\/(\s|$)/,  // recursive 777 on root
  /^dd\s+.*of=\/dev\/[sh]d/,       // writing to raw disk devices
  /^mkfs\./,                        // formatting filesystems
]

function isHardcodedBlocked(command: string): boolean {
  return HARDCODED_BLOCKS.some(pattern => pattern.test(command.trim()))
}
```

Layer 1 exists because some operations are catastrophic and irreversible regardless of context. If the model somehow ends up executing `rm -rf /`, no amount of user confirmation would make that acceptable — so it never even gets to the confirmation dialog.

### Layer 2 — Permission Configuration

The settings file and CLI flags configure allowlists and blocklists for commands and file paths:

```typescript
interface PermissionConfig {
  // Commands always allowed (skip layers 3-5)
  allowedCommands: string[]       // e.g., ['git status', 'npm test']

  // Commands always blocked (even if user confirms)
  blockedCommands: string[]       // e.g., ['curl', 'wget']

  // File paths the agent can read/write without confirmation
  allowedPaths: string[]          // e.g., ['/home/user/projects/']

  // File paths always blocked
  blockedPaths: string[]          // e.g., ['/etc/', '/usr/bin/']
}
```

This layer handles the "I trust this in my project directory" case. A developer who runs Claude Code in `~/projects/` might configure that directory as an allowed path, reducing confirmation prompts for normal development operations while still blocking access to system directories.

### Layer 3 — Tool-Level Permission Check

As covered in Chapter 3, each tool implements `needsPermission()`:

```typescript
// BashTool — dynamic permission based on command content
needsPermission(input: { command: string }): boolean {
  if (isHardcodedBlocked(input.command)) return true   // Will be blocked at L1 anyway
  if (isAllowlisted(input.command)) return false        // L2 cleared it
  if (isReadOnlyCommand(input.command)) return false    // ls, cat, grep: no confirm needed
  return true                                           // Everything else: needs confirm
}
```

Layer 3 is the intelligence layer — it applies semantic understanding of what the operation actually does, not just pattern matching. A command that reads a file doesn't need confirmation even if it's not on the allowlist. A command that modifies state does.

### Layer 4 — User Confirmation Dialog

When layers 1-3 don't resolve the question, the system presents a confirmation prompt:

```
⚠  Claude wants to run:
   git push origin main --force

   This will overwrite remote history on the main branch.
   [Y] Allow  [N] Deny  [A] Allow all similar  [S] Skip  [?] Explain
```

The prompt shows the exact operation, a risk summary, and multiple response options. "Allow all similar" creates a session-scoped rule that pre-approves similar operations for the rest of the session, reducing interruptions for repetitive tasks.

### Layer 5 — The YOLO Classifier

The YOLO ("You Only Live Once") classifier is an AI model — a smaller, faster model than the main agent — that independently assesses whether an operation is safe:

```typescript
interface YOLOClassifierResult {
  safe: boolean
  confidence: number       // 0.0 - 1.0
  rationale: string        // Why it made this assessment
  category: SafetyCategory // 'read-only' | 'reversible' | 'irreversible' | 'destructive'
}

async function runYOLOClassifier(
  operation: PendingOperation
): Promise<YOLOClassifierResult> {
  return classifierModel.classify({
    operation: operation.description,
    context: operation.conversationContext,
    criteria: YOLO_SAFETY_CRITERIA,
  })
}
```

The classifier provides a second opinion, independent of the tool's own `needsPermission()` assessment. It's particularly useful for novel operations that the tool's heuristics didn't anticipate.

## The Racing Mechanism — Concurrency as Safety Infrastructure

The architectural insight is that Layer 4 (user confirmation) and Layer 5 (classifier) run concurrently:

```typescript
// Permission racing
async function resolvePermission(operation: PendingOperation): Promise<PermissionResult> {
  // Race: user confirmation vs classifier assessment
  const result = await Promise.race([
    waitForUserConfirmation(operation),    // Layer 4
    runYOLOClassifier(operation),          // Layer 5
  ])

  if (result.source === 'classifier' && result.safe && result.confidence > 0.95) {
    // Classifier is very confident this is safe — proceed without user wait
    cancelUserDialog()
    return { allowed: true, source: 'classifier' }
  }

  if (result.source === 'user') {
    // User responded — classifier result no longer needed
    cancelClassifier()
    return { allowed: result.allowed, source: 'user' }
  }

  // Classifier returned low confidence or unsafe — wait for user
  return waitForUserConfirmation(operation)
}
```

The racing architecture means:
- If the classifier is confident an operation is safe, it can approve before the user even sees the dialog — zero latency overhead.
- If the classifier flags something unsafe, the user dialog remains but now includes the classifier's assessment.
- If the user responds before the classifier finishes, the classifier is cancelled.

This is safety without paralysis. The system doesn't force a round-trip to the user for every operation that needs checking — it uses intelligence to pre-approve safe ones and interrupt only when necessary.

## Irreversible Operations — Extra Scrutiny

Operations categorized as irreversible get special treatment:

```typescript
const IRREVERSIBLE_INDICATORS = [
  'delete', 'remove', 'drop', 'purge',    // Data deletion
  'push --force', 'reset --hard',          // History rewriting
  'publish', 'deploy', 'release',          // External mutations
  'send', 'email', 'webhook',              // External communications
]

function isIrreversible(operation: PendingOperation): boolean {
  return IRREVERSIBLE_INDICATORS.some(indicator =>
    operation.description.toLowerCase().includes(indicator)
  )
}

// For irreversible operations: require explicit user confirmation
// even if YOLO classifier thinks it's safe
if (isIrreversible(operation)) {
  return waitForUserConfirmation(operation)  // No racing for irreversible
}
```

Irreversible operations bypass the racing mechanism. The YOLO classifier can still run for its risk assessment, but user confirmation is required regardless of the classifier's verdict. The asymmetry is deliberate: false negatives (blocking a safe operation) are recoverable; false positives (allowing a destructive operation) may not be.

## Key Takeaways

- The five-layer stack addresses different failure modes: catastrophic errors (L1), policy (L2), tool intelligence (L3), human judgment (L4), and AI second opinion (L5).
- Hardcoded safety at Layer 1 is unconditional — no configuration, user input, or model output can override it.
- The racing mechanism runs user confirmation and the YOLO classifier concurrently — first to return a clear verdict wins, minimizing latency.
- Irreversible operations bypass the race and always require explicit user confirmation, regardless of classifier confidence.
- The system is designed to be fast for safe operations and slow (appropriately) only for genuinely risky ones — friction proportional to risk.
