# 110+ Strings → One System Prompt

> SYSTEM_PROMPT_DYNAMIC_BOUNDARY splits static (globally cached) from dynamic (per-session) sections — the first half costs nothing after the first request; the second half costs every time.

## Overview

The system prompt the model receives isn't written by hand and stored in a config file. It's assembled at runtime from over ten distinct sources: base instructions, project-specific context from `CLAUDE.md` files, tool definitions, environment metadata, active file context, memory, and more. By the time the model sees it, what was 110+ separate strings has become a single coherent instruction document.

`context.ts` is responsible for this assembly. But the sophistication isn't just in what gets included — it's in how it's structured. The `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` marker divides the prompt into two halves with a critical property: everything above the boundary is identical across sessions and can be served from Anthropic's prompt cache. Everything below varies per session and must be processed fresh.

Understanding this mechanism matters enormously for production agent costs. An uncached system prompt of 50,000 tokens costs roughly 10x more per request than a cached one. At any meaningful request volume, the difference is substantial.

## The Ten Sources — What Gets Assembled

The system prompt assembly pulls from these sources, in roughly this order:

```typescript
// context.ts — simplified assembly pipeline
async function buildSystemPrompt(session: Session): Promise<string> {
  const parts: string[] = []

  // === STATIC SECTION (above SYSTEM_PROMPT_DYNAMIC_BOUNDARY) ===

  // 1. Base system prompt — core identity and behavioral instructions
  parts.push(BASE_SYSTEM_PROMPT)

  // 2. Tool descriptions — all 40+ tools with their schemas
  parts.push(renderToolDescriptions(session.tools))

  // 3. User-level CLAUDE.md — from ~/.claude/CLAUDE.md
  const userMemory = await readClaudemd(HOME_DIR)
  if (userMemory) parts.push(userMemory)

  // 4. SYSTEM_PROMPT_DYNAMIC_BOUNDARY marker
  parts.push(SYSTEM_PROMPT_DYNAMIC_BOUNDARY)

  // === DYNAMIC SECTION (below boundary, fresh each request) ===

  // 5. Project-level CLAUDE.md — from current working directory
  const projectMemory = await readClaudemd(session.cwd)
  if (projectMemory) parts.push(projectMemory)

  // 6. Parent directory CLAUDE.md files — walk up the tree
  const parentMemory = await readParentClaudemds(session.cwd)
  parts.push(...parentMemory)

  // 7. Environment info — OS, shell, cwd, git status
  parts.push(buildEnvironmentInfo(session))

  // 8. Active file context — if user has files open/mentioned
  if (session.activeFile) parts.push(formatActiveFile(session.activeFile))

  // 9. MCP server context — connected servers and their capabilities
  parts.push(renderMCPContext(session.mcpClients))

  // 10. Session-specific injections — from commands, hooks, etc.
  parts.push(...session.systemInjections)

  return parts.join('\n\n')
}
```

The ordering is intentional: stable content first (base prompt, tool descriptions), then the boundary, then everything that varies by project and session.

## SYSTEM_PROMPT_DYNAMIC_BOUNDARY — Cache Economics

Anthropic's prompt caching works on a prefix match basis: if the first N tokens of a request's system prompt exactly match a previous request's system prompt, those tokens are served from cache at roughly 10% of the normal input token cost.

The `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` constant is a literal string marker that creates a predictable split point:

```typescript
const SYSTEM_PROMPT_DYNAMIC_BOUNDARY =
  '\n\n---DYNAMIC_CONTENT_BELOW_THIS_LINE---\n\n'
```

Everything above this marker must be identical across all requests for the cache to activate. Tool descriptions, base instructions, and user-level memory are good candidates — they change rarely. Project-specific context, current environment info, and session injections change constantly — they belong below the boundary.

```
Request 1 (cold):
  [Static section: 40,000 tokens] ← processed fresh, cached
  [Dynamic section: 5,000 tokens] ← processed fresh
  Cost: 45,000 tokens

Request 2 (warm):
  [Static section: 40,000 tokens] ← served from cache (90% discount)
  [Dynamic section: 5,000 tokens] ← processed fresh
  Cost: 4,000 tokens (effectively) + 5,000 tokens = 9,000 tokens

Savings: 36,000 tokens per request after warmup
```

For a busy developer running dozens of Claude Code sessions per day, this adds up to significant real dollar savings.

## CLAUDE.md Hierarchy — Project → Parent → Home

The `CLAUDE.md` file isn't just one file — it's a hierarchy:

```
/home/user/
├── .claude/
│   └── CLAUDE.md          ← User-level: always included (static section)
└── Projects/
    └── myproject/
        ├── CLAUDE.md       ← Project-level: included when cwd is here
        ├── src/
        │   └── CLAUDE.md   ← Subdirectory: included when cwd is here
        └── ...
```

The resolution algorithm:

```typescript
async function readAllClaudemds(cwd: string): Promise<string[]> {
  const results: string[] = []
  let dir = cwd

  // Walk up from cwd to filesystem root
  while (dir !== path.dirname(dir)) {
    const claudemd = path.join(dir, 'CLAUDE.md')
    if (await exists(claudemd)) {
      results.unshift(await readFile(claudemd))  // Prepend (parent before child)
    }
    dir = path.dirname(dir)
  }

  // Also check ~/.claude/CLAUDE.md
  const userClaudemd = path.join(HOME, '.claude', 'CLAUDE.md')
  if (await exists(userClaudemd)) {
    results.unshift(await readFile(userClaudemd))
  }

  return results
}
```

All found files are merged into the prompt. A project-level `CLAUDE.md` can override or supplement user-level instructions. Subdirectory `CLAUDE.md` files add context when working in that part of the codebase.

The user-level `CLAUDE.md` goes into the static section (it changes rarely), while project and subdirectory files go into the dynamic section (they change per project and per working directory).

## Environment Info — What Gets Injected

The environment section is generated fresh each session:

```typescript
function buildEnvironmentInfo(session: Session): string {
  return `
## Environment
- OS: ${process.platform} (${os.release()})
- Shell: ${session.shell}
- Working directory: ${session.cwd}
- Git branch: ${session.gitBranch ?? 'not a git repo'}
- Node version: ${process.version}
- Date/time: ${new Date().toISOString()}
`.trim()
}
```

This section is small but important — it tells the model what environment it's operating in, which affects how it interprets file paths, shell commands, and project structure.

## Key Takeaways

- The system prompt is assembled from 10+ sources at runtime, not stored as a single static string.
- `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` is the most important optimization: it creates a stable prefix that activates Anthropic's prompt cache, reducing per-request token costs by up to 90% for the static portion.
- The `CLAUDE.md` hierarchy (user → project → subdirectory) allows layered context: global preferences, project-specific instructions, and directory-specific guidance all merge seamlessly.
- Tool descriptions are part of the static (cached) section — updating a tool definition invalidates the cache for all sessions until the new version warms up.
- Environment information (cwd, git branch, timestamp) is always dynamic and always below the boundary — it changes every session and cannot be cached.
