# One Claude Commanding Many

> Fork subagents share the parent's prompt cache for near-zero-cost context inheritance — turning process spawning into a scalable orchestration primitive.

## Overview

Most people imagine multi-agent systems as something you build with a framework: LangGraph graphs, CrewAI crews, AutoGen conversations. Claude Code takes a different approach. It uses the tools that have been available since Unix 1969 — process spawning and file I/O — to run multiple Claude instances in parallel.

The pattern is called **coordinator mode**. One parent Claude process acts as the orchestrator. It reads the task, breaks it into parallel work units, spawns worker subprocess Claudes, waits for results written to temp files, then synthesizes everything into a coherent response. No framework. No message broker. No shared memory. Just processes and files.

What makes this worth understanding is not just its simplicity — it's the performance insight hiding inside it. Forked worker processes inherit the parent's prompt cache state, which means every worker starts with the full shared context already loaded at near-zero token cost.

## Coordinator Mode Architecture

The entry point is `coordinatorMode.ts`. Its job is to act as the conductor: it receives the original task, decides how to decompose it, and manages the lifecycle of worker processes.

```
┌─────────────────────────────────────────┐
│           Coordinator Claude            │
│  (reads task, plans decomposition)      │
└─────────┬─────────────┬────────────────┘
          │ spawn        │ spawn
          ▼             ▼
  ┌───────────┐  ┌───────────┐  ┌───────────┐
  │ Worker 1  │  │ Worker 2  │  │ Worker N  │
  │ (subtask) │  │ (subtask) │  │ (subtask) │
  └─────┬─────┘  └─────┬─────┘  └─────┬─────┘
        │               │               │
        ▼               ▼               ▼
  /tmp/result-1   /tmp/result-2   /tmp/result-N
          │               │               │
          └───────────────┴───────────────┘
                          │
                  Coordinator reads,
                  synthesizes, responds
```

The coordinator can run 10 or more workers simultaneously. For tasks like "analyze all files in this directory," it can assign one worker per file and process everything in parallel — something a single-Claude sequential approach cannot match.

## File-Based IPC

The inter-process communication mechanism is deliberately dumb: workers write their results to temporary files, and the coordinator reads those files when all workers complete.

```typescript
// Coordinator spawns a worker
const worker = spawn('claude', ['--worker', '--task-file', taskPath], {
  env: { ...process.env, WORKER_OUTPUT: outputPath }
});

// Worker writes result when done
await fs.writeFile(process.env.WORKER_OUTPUT, JSON.stringify(result));

// Coordinator collects all results
const results = await Promise.all(
  outputPaths.map(p => fs.readFile(p, 'utf8').then(JSON.parse))
);
```

This design has a key advantage: it is completely stateless between coordinator and workers. There is no shared memory to corrupt, no message queue to deadlock, no network call to fail. If a worker crashes, the coordinator just sees a missing output file and can handle it accordingly.

It also makes the whole system observable. You can inspect the temp files mid-execution to see what each worker has produced so far — something impossible with in-process function calls.

## Prompt Cache Inheritance

This is where the architecture gets clever. When the coordinator spawns workers as subprocesses via `fork()`, the child processes inherit the parent's memory state — including Anthropic's prompt cache state.

Claude Code's base system prompt is large: tool definitions, safety rules, behavior instructions. Loading that context into each worker from scratch would cost thousands of tokens per worker and significant latency. Instead, because workers are forked from the parent, they start with that context already in cache.

```
Parent process: [system prompt cached] [task context cached]
                        │
              ┌─────────┼──────────┐
              ▼         ▼          ▼
           Worker1   Worker2    Worker3
        [cache hit] [cache hit] [cache hit]  ← near-zero cost
        + [subtask] + [subtask] + [subtask]  ← only new tokens charged
```

This is the architectural reason Claude Code uses process forking rather than independent subprocess launches. Independent `spawn()` calls would each start cold. `fork()` starts warm.

## vs. Framework-Based Orchestration

LangGraph and CrewAI require you to model your multi-agent system as a graph or a team with defined roles and communication protocols. That's powerful for complex, well-defined workflows, but it's overhead for the common case.

Claude Code's approach trades expressiveness for simplicity:

| Aspect | Framework (LangGraph/CrewAI) | Claude Code approach |
|---|---|---|
| Coordination model | Graph / role-based | Process tree |
| IPC | Message passing / memory | File system |
| Setup | Schema + config | Just spawn |
| Debuggability | Framework tooling | `cat /tmp/result-*` |
| Prompt cache | Explicit management | Inherited via fork |

For a coding assistant that needs to parallelize tool execution and file analysis, the simple approach wins. The framework overhead would show up as latency without adding capability.

## Parallelism in Practice

The coordinator's power is raw throughput. Where a sequential agent processes tasks one at a time, a coordinator with 10 workers can process 10 tasks in the time it takes to process one — bounded only by the rate limits of the underlying API.

For Claude Code, common parallelization patterns include analyzing multiple files simultaneously, running multiple search operations concurrently, and distributing long-horizon subtasks across workers that each operate independently.

## Key Takeaways

- Coordinator mode uses process spawning and file I/O — not orchestration frameworks — to run multiple Claude instances in parallel
- Workers are forked from the parent process, inheriting prompt cache state for near-zero-cost context loading
- File-based IPC is stateless, observable, and crash-resilient compared to shared memory or message queues
- The coordinator can run 10+ workers simultaneously, offering linear throughput scaling for parallelizable tasks
- Simplicity is the design choice: no framework overhead, just Unix primitives applied to LLM orchestration
