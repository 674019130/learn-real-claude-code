# React in the Terminal

> Three interning pools (Char/Style/Hyperlink) turn O(n) string comparisons into O(1) integer checks — the performance trick that makes a React-based terminal feel instant.

## Overview

When you watch Claude Code stream output into your terminal — the spinner, the file trees, the syntax-highlighted code blocks — you are watching a React application render. Not React in a browser. React in a terminal emulator, where the "DOM" is a grid of character cells and the "CSS" is ANSI escape sequences.

This is made possible by Ink, a library that implements a custom React reconciler targeting terminal output instead of browser DOM. The idea sounds exotic until you realize that React's reconciler is deliberately designed to be renderer-agnostic. React Native uses a different reconciler targeting native mobile views. Ink uses one targeting terminal cells. Same React model, different output target.

Understanding how Ink works tells you something important about why Claude Code's terminal UI is as responsive as it is under heavy streaming load.

## The Ink Stack

Ink sits on top of two key dependencies beyond React itself:

```
┌─────────────────────────────────────────┐
│              Your React Code             │
│     <Box><Text color="green">Hi</Text>   │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│               Ink Reconciler            │
│   (translates React tree → cell grid)   │
└──────────────────┬──────────────────────┘
                   │
        ┌──────────┴──────────┐
        ▼                     ▼
┌───────────────┐    ┌────────────────────┐
│ Yoga Layout   │    │  Terminal Renderer  │
│ (flexbox for  │    │ (ANSI escape codes, │
│  terminal)    │    │  double buffer)     │
└───────────────┘    └────────────────────┘
```

**Yoga** is Facebook's C++ layout engine that implements CSS flexbox. It's the same engine React Native uses. By embedding Yoga, Ink gets proper flexbox layout in the terminal — you can use `flexDirection`, `justifyContent`, `alignItems` on terminal boxes and they behave exactly as they would in a browser. This is not a toy approximation; it's the real flexbox implementation.

## The Custom Reconciler

React's reconciler is the part that manages the virtual DOM diff and decides what updates to apply. When you write `<Text>hello</Text>` in Ink, the reconciler doesn't call `document.createElement` — it calls Ink's custom host functions that create entries in a cell grid.

```typescript
// React's reconciler host config (simplified Ink version)
const hostConfig = {
  createInstance(type, props) {
    return new InkNode(type, props);  // Not a DOM node
  },
  appendChild(parent, child) {
    parent.children.push(child);      // Custom tree
  },
  commitUpdate(instance, updatePayload, oldProps, newProps) {
    instance.update(newProps);        // Terminal cell update
  }
};
```

The reconciler is what makes Ink feel like React — you get hooks, state, effects, context, and the rest of the React programming model. You just get them in the terminal.

## Double-Buffered Screen Rendering

Ink uses a double-buffered rendering approach. At any given moment there are two representations of the terminal screen:

- **Back buffer**: where the new frame is assembled
- **Front buffer**: what is currently displayed

When a render cycle completes, Ink diffs the two buffers and emits only the ANSI escape sequences needed to update the terminal from old state to new state. If 90% of the screen didn't change, 90% of the screen gets zero escape sequences. This is analogous to how game engines use double buffering to avoid screen tearing, applied to character-cell displays.

```
Back buffer (new frame):
  row 0: "Hello world    "
  row 1: "Status: ██████ "

Front buffer (current display):
  row 0: "Hello world    "
  row 1: "Status: █████  "  ← only this cell changed

Diff output: cursor-to(1,14) + "█"  ← minimal escape sequence
```

This is critical for streaming performance. Claude Code streams tokens continuously. Without differential updates, every new token would redraw the entire screen — creating flicker and consuming I/O bandwidth proportional to screen size rather than change size.

## The Three Interning Pools

This is where Ink's performance design gets precise. Every character cell in the terminal grid has three attributes: its character, its style (color, bold, italic, etc.), and optionally a hyperlink. Naively, comparing two cells means comparing strings — O(n) where n is string length.

Ink uses interning pools to convert these strings into integer IDs:

```
Char pool:    "█" → 1042,  "H" → 72,   " " → 32
Style pool:   "color:green;bold" → 7,   "color:red" → 3
Hyperlink pool: "https://..." → 99,     null → 0

Cell comparison:
  Old: (charId=1042, styleId=7, linkId=0)
  New: (charId=1042, styleId=7, linkId=0)
  Equal? → 1042===1042 && 7===7 && 0===0  ← O(1) integer compare
```

Instead of comparing `"█" === "█"` (string comparison, O(n) in character length) or `"color:green;bold" === "color:green;bold"` (much longer), the comparison becomes integer equality — three integer comparisons per cell, regardless of string complexity.

For a terminal running at 80×24 = 1920 cells, this means 1920 × 3 integer comparisons per render cycle instead of 1920 potentially-long string comparisons. At streaming speed, this accumulates into a meaningful performance difference.

## Why This Matters for Claude Code

Claude Code streams responses continuously, updating the terminal UI at high frequency. Every streamed token potentially changes the display. Without the optimizations in Ink — double buffering, interning pools, differential escape output — the UI would be visibly sluggish or flickery under load.

The architecture shows a common pattern in performance-critical systems: the expensive part is not the computation, it's the comparison. By interning strings into IDs up front, comparison cost collapses from O(n) to O(1) — and that savings multiplies across every cell, every frame, every second of streaming.

## Key Takeaways

- Ink implements a custom React reconciler that targets terminal cell grids instead of browser DOM
- Yoga provides real CSS flexbox layout for terminal components — the same engine React Native uses
- Double buffering ensures only changed cells emit ANSI escape sequences, keeping I/O minimal during streaming
- Three interning pools (Char, Style, Hyperlink) convert string attributes to integer IDs for O(1) cell comparison
- The result is a React programming model — hooks, state, effects — that renders to the terminal at streaming speed
