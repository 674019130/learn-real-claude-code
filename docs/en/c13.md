# Dual Feature Gating

> Bun's `feature()` eliminates code at compile time; GrowthBook/Statsig gates features at runtime — two systems, complementary roles, neither replaceable by the other.

## Overview

Claude Code ships to millions of users across different contexts: CLI users, enterprise deployments, a coordinator mode for multi-agent workflows, and internal builds with unreleased features. Each of these targets needs a different feature set. How do you maintain one codebase while producing four or more meaningfully different products from it?

The answer is a dual-layer feature gating system. The first layer uses Bun's compile-time `feature()` function to eliminate code at bundle time — features that don't belong in a build are not just disabled, they don't exist in the binary. The second layer uses GrowthBook and Statsig to control feature availability at runtime, enabling gradual rollouts and A/B experiments without shipping new code.

Understanding both layers reveals why 108 modules are imported in the source but not present in any given build — and why that's intentional, not an error.

## Compile-Time Gating: Bun's `feature()`

Bun's bundler supports a `feature()` function that evaluates at bundle time. Any code path gated by a feature flag that evaluates to `false` is dead-code-eliminated from the output — the same way `if (false) { ... }` is eliminated by any optimizing compiler.

```typescript
// In source code
if (feature('KAIROS_MODE')) {
  import('./kairos/backgroundAgent');
  initializeKairosMonitor();
}

if (feature('BUDDY_SYSTEM')) {
  import('./buddy/companion');
  spawnBuddyIfEligible(userId);
}
```

When building the standard CLI, `KAIROS_MODE` and `BUDDY_SYSTEM` evaluate to `false`. The bundler eliminates both branches entirely. The output binary contains no trace of KAIROS or BUDDY code — not just unreachable, literally absent.

This produces meaningfully different builds from the same source:

```
┌──────────────────────────────────────────────────────────┐
│                    Source Codebase                        │
│  [All features, all modes, all targets]                  │
└────────────┬───────────┬──────────────┬──────────────────┘
             │           │              │
      ┌──────▼──┐  ┌─────▼──┐  ┌───────▼──────┐
      │  CLI    │  │ Bridge │  │  Coordinator  │
      │  Build  │  │ Build  │  │  Build        │
      │ (std)   │  │ (IDE)  │  │ (multi-agent) │
      └─────────┘  └────────┘  └───────────────┘
```

Each build gets only the code relevant to its function. The Bridge build (for IDE integrations like VS Code) includes the IDE communication layer. The Coordinator build includes multi-agent orchestration. The standard CLI includes neither.

## The 108 Missing Modules

A source analysis of Claude Code reveals approximately 108 modules that are imported in the TypeScript source but are absent from any available build. These are stub modules for features that have not shipped yet — enterprise capabilities, unreleased integrations, future functionality currently under development.

```typescript
// Source references this module
import { EnterpriseAuthProvider } from './enterprise/auth-provider';

// But './enterprise/auth-provider' doesn't exist in any shipped build
// Bun eliminates the import path at bundle time for non-enterprise builds
// Enterprise builds would include the actual implementation
```

This is not a bug or an oversight. It is the compile-time gating system working as designed. The source code acts as the unified specification; each build configuration determines which implementations get included. The 108 missing modules represent the gap between what the source declares and what any given build ships.

For developers studying the codebase, these stubs are a roadmap of planned and in-progress features. For security, it means enterprise-only code is not present in consumer builds — it cannot be enabled by any runtime configuration change because the code does not exist in the binary.

## Runtime Gating: GrowthBook and Statsig

Compile-time gating is binary: a feature is either in a build or it isn't. Runtime gating adds a third dimension — features that are in a build but enabled only for a subset of users.

Claude Code integrates both GrowthBook and Statsig for different runtime gating scenarios:

```typescript
// GrowthBook for feature flags and experiments
const gb = new GrowthBook({ userId, attributes });
if (gb.isOn('new-streaming-ui')) {
  renderNewStreamingUI();
} else {
  renderLegacyUI();
}

// Statsig for event tracking + flag evaluation
const inExperiment = await statsig.checkGate('parallel-tool-execution');
if (inExperiment) {
  runToolsInParallel();
}
```

**GrowthBook** handles A/B testing and feature flags with a focus on experiment analysis. It evaluates flags client-side against rules that can target by user attributes — allowing percentage rollouts, cohort targeting, and experiment assignment.

**Statsig** handles both flag evaluation and telemetry, making it easy to correlate feature exposure with downstream metrics. If a new feature affects session length or error rates, Statsig captures both the exposure and the outcome in a unified system.

## The Two Systems Are Complementary

It would be wrong to think of compile-time and runtime gating as redundant. They solve different problems:

| Dimension | Compile-time (Bun) | Runtime (GrowthBook/Statsig) |
|---|---|---|
| When decided | At build time | At execution time |
| Granularity | Whole build targets | Individual users |
| Changeability | Requires new build | Config push |
| Security | Code absent from binary | Code present, gated |
| Use case | Architecture splits | Gradual rollouts |

You cannot use runtime flags to exclude code from a security-sensitive build — the code exists in the binary and could be enabled. You cannot use compile-time flags for gradual rollouts — that would require shipping a new build for every percentage point of rollout. The two systems are not alternatives; they operate at different layers of the stack.

## Key Takeaways

- Bun's `feature()` function performs dead-code elimination at bundle time, producing meaningfully different binaries (CLI, Bridge, Coordinator, KAIROS) from one source
- The ~108 missing modules in the source are stub imports for enterprise and unreleased features — eliminated at compile time for non-enterprise builds
- GrowthBook handles A/B experiments and feature flags; Statsig handles flag evaluation plus telemetry correlation
- Compile-time gating secures architecture splits (code absent from binary); runtime gating enables gradual rollouts (code present, user-targeted)
- The dual system is not redundant — each layer handles the cases the other cannot
