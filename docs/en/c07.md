# Memory & Dreams

> Auto-Dream runs four phases (orient → gather → consolidate → prune) with a read-only bash subagent — the agent literally sleeps and restructures its own memory.

## Overview

Sessions end. Conversations reset. The context window, no matter how well managed, doesn't persist between runs. Most agent systems accept this as a fundamental limitation. Claude Code doesn't.

Cross-session memory works through `CLAUDE.md` files — structured markdown files written to disk that are loaded into the system prompt on every new session. This is the persistence layer. But simply appending facts to a file over time creates a different problem: memory bloat. A file that grows without pruning becomes too long to be useful, and eventually too long to fit in the system prompt.

The solution is Auto-Dream: a four-phase memory consolidation process that runs when instructed, using a read-only bash subagent to read existing memory files, gather relevant facts from the current conversation, merge and deduplicate memories, and prune stale entries. It's architecturally notable because the agent uses itself — in a constrained form — to restructure its own persistent state.

## The Persistence Layer — CLAUDE.md as Memory

The mechanism is elegantly simple: write important information to a file; load that file on startup. The `CLAUDE.md` hierarchy (covered in Chapter 5) means memory can exist at multiple scopes:

```
~/.claude/CLAUDE.md              ← Global memory: user preferences, patterns
~/Projects/myapp/CLAUDE.md       ← Project memory: architecture decisions, conventions
~/Projects/myapp/src/CLAUDE.md   ← Directory memory: specific file context
```

The memory format is structured markdown, not a database:

```markdown
# Memory

## User Preferences
- Prefers TypeScript over JavaScript
- Uses vitest for testing, not jest
- Dislikes verbose error messages — keep them terse

## Project State (myapp)
- Last worked on: authentication refactor (2026-03-15)
- Known issue: race condition in token refresh (see src/auth/tokenRefresh.ts:247)
- Decision: using Drizzle ORM, not Prisma — migration in progress

## Active Context
- Interviewing for senior positions — focus on system design explanations
```

This format is human-readable (the user can edit it directly), machine-readable (the model can parse it), and version-controlled (it's a markdown file in a git repo).

## The Four Phases of Auto-Dream

When the user (or a scheduled trigger) initiates memory consolidation, the Auto-Dream process runs:

### Phase 1: orient

The subagent reads all existing memory files to understand current state:

```typescript
// Phase 1: orient — build a map of existing memories
async function orientPhase(memoryPaths: string[]): Promise<MemoryMap> {
  const existing: MemoryMap = {}

  for (const path of memoryPaths) {
    if (await fileExists(path)) {
      existing[path] = await readFile(path)
    }
  }

  return existing
}
```

The orient phase establishes a baseline. The agent knows what it already knows before it tries to add anything new. Without this phase, consolidation would blindly append, creating duplicates.

### Phase 2: gather

The subagent analyzes the current conversation to extract facts worth remembering:

```typescript
// Phase 2: gather — extract important facts from conversation
async function gatherPhase(
  conversation: Message[],
  existingMemory: MemoryMap
): Promise<ExtractedFacts[]> {

  // Invoke model to identify what's worth persisting
  const extractionPrompt = buildExtractionPrompt(conversation, existingMemory)
  const response = await callModel(extractionPrompt)

  return parseExtractedFacts(response)
}

// What the extraction prompt asks for:
const EXTRACTION_CRITERIA = `
Identify facts worth persisting across sessions:
- User preferences and working style patterns
- Project architecture decisions and their rationale
- Known issues and their locations in the codebase
- Conventions established during this session
- Context needed to resume work effectively

Do NOT capture:
- Specific file contents (too volatile)
- Step-by-step task execution logs
- Temporary states that will change
`
```

The criteria are explicit about what to extract and what to skip. Capturing file contents would bloat the memory quickly and become stale. Capturing the fact that "src/auth/tokenRefresh.ts has a known race condition" is durable and useful.

### Phase 3: consolidate

The subagent merges newly gathered facts with existing memories, deduplicating and updating stale entries:

```typescript
// Phase 3: consolidate — merge and deduplicate
async function consolidatePhase(
  existing: MemoryMap,
  newFacts: ExtractedFacts[]
): Promise<MemoryMap> {

  const consolidationPrompt = buildConsolidationPrompt(existing, newFacts)
  const consolidated = await callModel(consolidationPrompt)

  // The model returns a new version of each memory file
  // with duplicates removed, contradictions resolved,
  // and new information integrated
  return parseConsolidatedMemory(consolidated)
}
```

Consolidation is the step that prevents memory bloat. If the existing memory says "prefers vitest" and the new conversation confirms the same, the consolidation keeps one clean entry. If the conversation reveals a preference has changed, the old entry is updated.

### Phase 4: prune

The final phase removes entries that are no longer relevant:

```typescript
// Phase 4: prune — remove stale memories
async function prunePhase(
  consolidated: MemoryMap,
  pruningRules: PruningConfig
): Promise<MemoryMap> {

  const pruned: MemoryMap = {}

  for (const [path, content] of Object.entries(consolidated)) {
    const prunePrompt = buildPrunePrompt(content, pruningRules)
    pruned[path] = await callModel(prunePrompt)
  }

  return pruned
}

// Pruning criteria:
const PRUNING_RULES = {
  // Remove memories older than this if not recently referenced
  maxAgeWithoutReference: '30 days',
  // Remove memories about completed one-time tasks
  removeCompletedTaskContexts: true,
  // Always keep user preferences regardless of age
  neverPruneCategories: ['User Preferences'],
}
```

Pruning is what makes the memory system sustainable over months of use. Without it, old project contexts, resolved issues, and completed task states would accumulate forever.

## The Read-Only Bash Subagent

A detail worth noting: during the dream phases, the subagent operates in a read-only mode. It can read files and call the model, but it cannot write to the filesystem directly. All writes happen at the end of the dream process, after all four phases have completed and the user has had a chance to review:

```
orient (reads memory files)
  ↓
gather (reads conversation, calls model)
  ↓
consolidate (calls model with merged content)
  ↓
prune (calls model to clean up)
  ↓
[review checkpoint — user can inspect before committing]
  ↓
write new memory files to disk
```

Read-only subagent execution prevents the partially-complete state problem: if the dream is interrupted, no memory has been corrupted. The original files remain intact.

## Memory Written as Structured Markdown

The final output is written as organized, timestamped markdown:

```markdown
# Memory

*Last consolidated: 2026-03-15 14:30*

## User Preferences
- TypeScript with strict mode preferred
- vitest over jest — already configured in all projects

## Project Context (myapp)
- **Architecture**: Next.js + Supabase + Tailwind v4
- **Known issue**: race condition in `src/auth/tokenRefresh.ts:247` — scheduled fix
- **In progress**: Drizzle ORM migration from Prisma (70% complete as of 2026-03-14)

## Conventions
- Commit messages: conventional commits format
- PR reviews: focus on type safety and edge cases
```

## Key Takeaways

- Cross-session persistence works through `CLAUDE.md` files — plain markdown on disk, loaded into the system prompt on every new session.
- Auto-Dream runs four phases: orient (read existing), gather (extract from conversation), consolidate (merge and deduplicate), prune (remove stale entries).
- The subagent operates read-only during all phases — writes happen only at the end, preventing corruption from interrupted consolidations.
- Memory is structured markdown, not a database — human-editable, version-controlled, and directly readable by both humans and the model.
- Pruning is what makes the system sustainable — without it, memory files would grow indefinitely and eventually become too large to be useful or even loadable.
