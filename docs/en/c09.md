# Hook Lifecycle

> Hooks can modify tool input, block execution, or inject system messages — three backends (command/HTTP/prompt) give you Unix pipes, webhooks, and prompt engineering in one unified system.

## Overview

Hooks are the extension mechanism for Claude Code's tool execution pipeline. They let you intercept, inspect, modify, or block tool calls at defined points in their lifecycle — without modifying Claude Code's source code. If you want to log every bash command to a centralized audit system, prevent file writes outside a sandboxed directory, or automatically append safety context to shell tool inputs, hooks are the mechanism.

The hook system defines three lifecycle points, three backend types, and a rich set of possible responses for each hook. Understanding it means understanding both how to build production safety tooling on top of Claude Code, and how Claude Code itself uses hooks internally for features like the YOLO classifier and permission racing.

## Three Hook Types — Where in the Lifecycle

```
User/Model initiates tool call
        │
        ▼
┌──────────────────────┐
│   PreToolUse Hook    │  ← Can BLOCK, MODIFY input, or pass through
└──────────────────────┘
        │
        ▼
┌──────────────────────┐
│   Tool Execution     │  ← The actual tool.invoke() call
└──────────────────────┘
        │
        ▼
┌──────────────────────┐
│  PostToolUse Hook    │  ← Can MODIFY result, APPEND system messages
└──────────────────────┘
        │
        ▼
┌──────────────────────┐
│    Stop Hook         │  ← Fires when agent loop ends
└──────────────────────┘
        │
        ▼
Result returned to model / session ends
```

### PreToolUse

Fires before `tool.invoke()` is called. Receives the full tool call including name and input parameters. Can return:

- **`continue`** — pass through unchanged
- **`block`** — prevent execution, return an error result to the model
- **`modify`** — change the input parameters before execution proceeds

```typescript
// PreToolUse hook response types
type PreToolUseResponse =
  | { action: 'continue' }
  | { action: 'block'; reason: string }
  | { action: 'modify'; newInput: Record<string, unknown> }
```

The modify action is particularly powerful. A hook can rewrite the tool's input parameters completely — adding flags to shell commands, canonicalizing file paths, injecting safety constraints.

### PostToolUse

Fires after `tool.invoke()` completes and `renderToolResult()` has run. Receives the tool name, original input, and rendered result. Can return:

- **`continue`** — pass through unchanged
- **`modify`** — change the rendered result the model will see
- **`inject`** — append a system message to the conversation alongside the tool result

```typescript
// PostToolUse hook response types
type PostToolUseResponse =
  | { action: 'continue' }
  | { action: 'modify'; newResult: ToolResultContent }
  | { action: 'inject'; systemMessage: string }
```

The inject action creates a parallel channel for hooks to communicate with the model. Rather than modifying the tool result itself, the hook adds context that the model will see but the user typically won't (unless they inspect the raw conversation).

### Stop

Fires when the agent loop exits — either by natural `end_turn`, by user interrupt, or by session end. Useful for cleanup, audit logging, or triggering downstream processes:

```typescript
type StopHookPayload = {
  reason: 'end_turn' | 'user_interrupt' | 'max_tokens' | 'error'
  sessionId: string
  summary: SessionSummary  // Token counts, tools used, etc.
}
```

## Three Backends — How Hooks Execute

### Backend 1: Shell Command

The most Unix-native backend. The hook is a shell command that receives the event payload via stdin (JSON) and returns a response on stdout (JSON):

```json
// settings.json — shell command hook
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "backend": "command",
        "command": "/usr/local/bin/audit-bash-command.sh"
      }
    ]
  }
}
```

```bash
#!/bin/bash
# audit-bash-command.sh
# Reads JSON from stdin, logs command, outputs continue/block

payload=$(cat)
command=$(echo "$payload" | jq -r '.input.command')

# Log to audit file
echo "$(date -u +%Y-%m-%dT%H:%M:%SZ) $command" >> /var/log/claude-code-commands.log

# Block dangerous patterns
if echo "$command" | grep -qE 'curl.*\|.*sh|wget.*\|.*sh'; then
  echo '{"action":"block","reason":"Pipe-to-shell pattern blocked by security policy"}'
  exit 0
fi

echo '{"action":"continue"}'
```

This backend integrates naturally with existing Unix tooling: grep for pattern matching, jq for JSON processing, your existing logging infrastructure.

### Backend 2: HTTP Webhook

For hooks that need to call external services — a centralized audit API, a policy service, an approval workflow:

```json
// settings.json — HTTP webhook hook
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "*",
        "backend": "http",
        "url": "https://policy.internal.example.com/check",
        "timeout": 2000,
        "headers": { "Authorization": "Bearer $POLICY_TOKEN" }
      }
    ]
  }
}
```

The hook service receives a POST with the event payload and must respond within the configured timeout. Timeout handling is critical: if a PreToolUse HTTP hook times out, the tool call proceeds by default (fail-open) or is blocked by default (fail-closed), depending on configuration.

### Backend 3: Prompt Injection

The most unusual backend — a hook that works by injecting text into the model's context rather than executing external code:

```json
// settings.json — prompt injection hook
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write",
        "backend": "prompt",
        "inject": "Before writing any file, consider whether this change should be part of a larger atomic commit. Check if related files need updates."
      }
    ]
  }
}
```

This backend injects a system message into the conversation at the hook point. The model reads it as context before proceeding. It's less deterministic than command or HTTP hooks (the model may or may not act on the injected text), but it's useful for soft guidance that influences behavior without hard blocking.

## Matcher Patterns

Hooks are configured with matchers that control which tools they apply to:

```json
{ "matcher": "Bash" }         // Only the Bash tool
{ "matcher": "Write" }        // Only the Write file tool
{ "matcher": "*" }            // All tools
{ "matcher": "Bash|Write" }   // Bash or Write
{ "matcher": "!Read" }        // All tools except Read
```

Matchers keep hooks scoped. An audit hook for shell commands shouldn't fire for file reads — not because it would break anything, but because the noise would obscure the signal.

## Async Patterns — Parallel Safe Hooks

Hooks that don't depend on each other's results run in parallel:

```typescript
// Multiple PreToolUse hooks on the same tool — run concurrently
async function runPreToolUseHooks(
  toolCall: ToolCall,
  hooks: PreToolUseHook[]
): Promise<PreToolUseResponse> {
  const matchingHooks = hooks.filter(h => matches(h.matcher, toolCall.name))

  // Run all matching hooks concurrently
  const responses = await Promise.all(
    matchingHooks.map(h => executeHook(h, toolCall))
  )

  // If any hook blocks, the operation is blocked
  const blockResponse = responses.find(r => r.action === 'block')
  if (blockResponse) return blockResponse

  // Merge modifications (last write wins for conflicting keys)
  const modifications = responses.filter(r => r.action === 'modify')
  return mergeModifications(modifications) ?? { action: 'continue' }
}
```

The "any block wins" policy for multiple hooks is correct for security: if one hook determines an operation is unsafe, it doesn't matter that another hook thinks it's fine. The conservative result wins.

## Real-World Hook Patterns

**Audit logging all tool calls:**
```bash
# PostToolUse shell hook on * matcher
echo "$payload" | jq '{ts: now, tool: .tool, input: .input, exitCode: .result.exitCode}' \
  >> ~/audit/claude-code.jsonl
echo '{"action":"continue"}'
```

**Restricting file writes to project directory:**
```bash
# PreToolUse shell hook on Write matcher
path=$(echo "$payload" | jq -r '.input.path')
if [[ "$path" != /home/user/projects/* ]]; then
  echo "{\"action\":\"block\",\"reason\":\"Write outside project directory blocked: $path\"}"
else
  echo '{"action":"continue"}'
fi
```

**Injecting safety reminders for network operations:**
```json
{
  "matcher": "Bash",
  "backend": "prompt",
  "inject": "If this command makes outbound network requests, confirm the target URL is expected for this task."
}
```

## Key Takeaways

- Three hook types cover the full tool lifecycle: PreToolUse (before execution), PostToolUse (after execution), Stop (when the session ends).
- PreToolUse hooks can block execution, modify input parameters, or pass through — modification allows transforming tool calls without blocking them.
- PostToolUse hooks can modify the rendered result or inject system messages alongside the result — parallel channels for different audiences.
- Three backends (command/HTTP/prompt) support Unix shell scripts, external microservices, and model context injection respectively.
- Multiple hooks on the same tool run concurrently; "any block wins" is the policy when multiple hooks disagree — conservative by design.
- Matchers (`*`, `Bash`, `!Read`) keep hooks scoped to relevant tools, reducing noise and improving debugging.
