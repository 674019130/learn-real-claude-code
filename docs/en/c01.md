# Booting 512K Lines in 500ms

> Fast paths exit before importing anything; expensive I/O runs in parallel with module evaluation — startup speed is an architectural decision, not a lucky accident.

## Overview

When you type `claude` and hit Enter, something happens that most developers never think about: a 512,000-line TypeScript codebase has to be parsed, evaluated, and ready to serve your first token in under half a second. Most tools fail this benchmark. Claude Code passes it — not by accident, but through deliberate architectural choices baked into its module structure.

This chapter dissects the cold-start path from the first line of `entrypoints/cli.tsx` to the moment the agent loop becomes live. Understanding this matters if you're building any long-lived agent CLI or persistent daemon — the same patterns apply whether your tool handles one request or thousands.

The key insight is that startup speed is entirely about *what you defer*. Every import has a cost. Every config read has a cost. Every network call has a cost. The craft lies in sequencing these costs so that nothing slow happens on the critical path to "ready."

## Fast Paths — Exit Before You Even Load

The very first code executed in `entrypoints/cli.tsx` is a handful of conditional checks:

```typescript
// entrypoints/cli.tsx (simplified)
const arg = process.argv[2]

if (arg === '--version' || arg === '-v') {
  console.log(VERSION)
  process.exit(0)
}

if (arg === '--help' || arg === '-h') {
  printHelp()
  process.exit(0)
}

// Only NOW do we import the main module
const { main } = await import('../src/main.tsx')
await main()
```

This is not an optimization — it's a philosophy. `--version` and `--help` save roughly **200ms** by exiting before a single feature module is evaluated. The Node.js module system is synchronous and expensive: every `import` triggers file reads, parsing, and execution. By placing the fastest exits at the top of the entry point, you guarantee they never pay for code they don't need.

Compare this to the naive approach, which many CLIs accidentally use: a top-level barrel file that imports everything, which then checks the flags. By then you've already paid the full startup cost.

## Three-Phase Lazy Loading

The real payload — `main.tsx` — is a monolith of roughly 804KB. It contains the agent loop, tool orchestration, UI rendering, and dozens of subsystems. Loading it synchronously would be catastrophic for startup time. Instead, Claude Code uses a three-phase lazy loading strategy:

```
Phase 1: CLI registers fast paths
         ↓
Phase 2: main.tsx is dynamically imported only after flag checks pass
         ↓
Phase 3: Feature modules inside main.tsx load lazily on first use
```

Dynamic `import()` in Node.js is non-blocking at the point of declaration. By using `await import('../src/main.tsx')` instead of a static top-level import, the entry point file itself stays tiny and fast to evaluate. The 804KB monolith is only loaded when you actually need it.

Inside `main.tsx`, the same principle repeats at the feature level. Tool implementations, specific command handlers, and optional subsystems are imported lazily behind conditional branches or on-demand factory functions — not at module initialization time.

## Parallel Prefetch — Concurrency at Boot

Once the main module is initialized, there's still a set of blocking operations required before the first user interaction: reading config files, validating authentication tokens, and initializing the MCP (Model Context Protocol) server connections.

Naive sequential approach:

```typescript
// Sequential — each step blocks the next
const config = await readConfig()          // 80ms
const auth = await validateAuth()          // 120ms
const mcpClients = await initMCP()         // 150ms
// Total: ~350ms just waiting
```

Production approach in Claude Code:

```typescript
// Parallel — all fire at the same time
const [config, auth, mcpClients] = await Promise.all([
  readConfig(),
  validateAuth(),
  initMCP(),
])
// Total: ~150ms (dominated by slowest single operation)
```

This concurrency at boot is crucial because config reads (disk I/O), auth checks (often network), and MCP initialization (socket connections) are all independent. There's no reason to serialize them. By firing them all concurrently while the main module is still evaluating its final setup code, the wall-clock time from "entry point executes" to "ready for first message" drops dramatically.

## The 804KB Monolith — Why It Exists

`main.tsx` being 804KB seems like a code smell — shouldn't that be split into smaller modules? In a traditional web application, yes. For an agent CLI that runs as a single long-lived process, there are trade-offs:

```
Separate small modules:
  + Easier to reason about individually
  - Multiple round-trips for module resolution
  - More file handles open simultaneously
  - Harder to share in-process state efficiently

Single monolith (with lazy sub-imports):
  + One parse, one evaluation, shared heap
  + Internal references resolve instantly (no module boundary crossing)
  - Must be disciplined about lazy-loading feature paths
```

The 804KB file represents a deliberate decision: pay the loading cost once, then operate with low overhead. This makes sense for an interactive CLI where startup is a one-time cost and the process runs for minutes or hours.

## Key Takeaways

- Fast-path flags (`--version`, `--help`) should exit before any feature imports run — this is worth 150-200ms on real hardware.
- Use `await import()` (dynamic import) for the main payload, keeping entry points tiny and cheap to evaluate.
- Boot-time I/O operations (config, auth, external service init) are almost always independent — run them with `Promise.all`, not sequentially.
- A large monolith with internal lazy-loading can outperform many small eagerly-loaded modules for long-lived processes, because module resolution overhead adds up.
- Treat startup time as an architectural constraint, not an afterthought — the patterns are simple but they must be in place from the beginning.
