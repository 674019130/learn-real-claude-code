# MCP: Infinite Extension

> 18 deferred tools load on-demand via ToolSearchTool to keep the base prompt under 200K tokens — the architecture that makes infinite extensibility practical.

## Overview

The Model Context Protocol (MCP) is Anthropic's answer to a fundamental scaling problem: how do you give an AI assistant access to hundreds of external tools without making every single request prohibitively expensive? The naive solution — load all tool definitions upfront — breaks down fast. Tool definitions are not free. Each one consumes tokens. Load enough of them and you exceed context limits before the conversation even starts.

Claude Code solves this with a deferred loading architecture. Of the tools available to Claude Code, 18 are "deferred" — their definitions are not included in the base prompt. They exist as stubs, identified only by name. When the model needs one, it calls a meta-tool called `ToolSearchTool` to fetch the full schema, and only then can it invoke the tool.

This is MCP's killer feature in practice: not just that you can add tools, but that the system is designed to scale to an unbounded number of tools without the context window paying for tools you never use.

## What MCP Actually Is

MCP is a standardized protocol for connecting language models to external capabilities. Before MCP, every tool integration was bespoke — different schemas, different transports, different error conventions. MCP defines a common interface so that any MCP-compatible server can expose tools that any MCP-compatible client (like Claude Code) can use.

The protocol supports three transport types:

```
┌─────────────────────────────────────────────────────┐
│                   Transport Types                    │
├──────────────┬──────────────────────────────────────┤
│ stdio        │ Subprocess communication via stdin/  │
│              │ stdout. Used for local tools.         │
├──────────────┬──────────────────────────────────────┤
│ SSE          │ HTTP Server-Sent Events for streaming │
│              │ responses from remote servers.        │
├──────────────┬──────────────────────────────────────┤
│ WebSocket    │ Bidirectional for tools that push     │
│              │ events back to the client.            │
└──────────────┴──────────────────────────────────────┘
```

From the model's perspective, all three look identical. A tool is a tool. The transport is an implementation detail the model never sees.

## Tool Bridging: Native vs. MCP

One of MCP's design goals is that external tools should be indistinguishable from native tools. When Claude Code calls `Bash` (a native tool) or calls a custom MCP tool you've defined, the model uses exactly the same calling syntax:

```json
{
  "name": "your_custom_tool",
  "input": {
    "parameter": "value"
  }
}
```

The host application (Claude Code) handles the dispatch — routing native calls internally and MCP calls to their respective servers. The model never knows the difference. This is what makes MCP powerful for extension: you're not adding a second-class tool category, you're adding first-class tools.

## The Deferred Loading System

Here is the architecture that makes large tool registries viable:

```
Base prompt loads:
  ├── Native tools (Bash, Read, Write, Edit, Glob, Grep...)
  ├── Core MCP tools (immediately needed)
  └── ToolSearchTool  ←── the meta-tool

Deferred (not in base prompt):
  ├── Tool stub 1 (name only)
  ├── Tool stub 2 (name only)
  ├── ...
  └── Tool stub 18 (name only)

When model needs a deferred tool:
  1. Calls ToolSearchTool("select:tool_name")
  2. Receives full JSON schema back
  3. Tool is now callable for the rest of the session
```

Without this pattern, loading 18 additional tool definitions — each potentially 500-2000 tokens of schema — would add 9,000 to 36,000 tokens to every single request. For a system already managing large context windows, that's the difference between fitting your task and hitting limits.

The deferred pattern is essentially lazy loading applied to the context window. Pay only for what you use.

## ToolSearchTool: The Meta-Tool

`ToolSearchTool` is worth examining closely because it represents a new kind of capability: a tool that gives the model access to more tools. It's self-referential in an interesting way.

```typescript
// The model calls this to discover and load a deferred tool
ToolSearch({
  query: "select:mcp__my_server__my_tool",
  max_results: 5
})

// Returns the full JSON schema for the matched tool(s)
// After this call, the model can invoke the tool directly
```

The search supports both exact selection (`select:tool_name`) and keyword search. This means the model can discover tools it doesn't know the exact name of — a form of tool introspection that wouldn't be possible if all schemas had to be defined upfront.

## Building Your Own MCP Server

The practical implication is that you can extend Claude Code with domain-specific tools without modifying anything in the core codebase. Define an MCP server, expose tools via the protocol, register it — and Claude Code can use your tools as if they were native.

```typescript
// Minimal MCP server (stdio transport)
import { Server } from "@modelcontextprotocol/sdk/server/index.js";

const server = new Server({ name: "my-tools", version: "1.0.0" });

server.setRequestHandler(ListToolsRequestSchema, async () => ({
  tools: [{
    name: "my_custom_tool",
    description: "Does something useful",
    inputSchema: {
      type: "object",
      properties: { input: { type: "string" } }
    }
  }]
}));

server.setRequestHandler(CallToolRequestSchema, async (request) => {
  // Handle tool execution
  return { content: [{ type: "text", text: "result" }] };
});
```

The server speaks MCP. Claude Code speaks MCP. They interoperate without either knowing anything specific about the other's implementation.

## Key Takeaways

- MCP is a standardized protocol for connecting LLMs to external tools across stdio, SSE, and WebSocket transports
- Tool bridging makes MCP tools indistinguishable from native tools at the model level
- 18 deferred tools in Claude Code load on-demand via ToolSearchTool — avoiding 9K–36K tokens of upfront schema cost
- ToolSearchTool is a meta-tool: it gives the model the ability to discover and load more tools at runtime
- Deferred loading is lazy evaluation applied to context windows — a pattern that scales to hundreds of tools without prohibitive token cost
