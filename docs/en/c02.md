# The Real Agent Loop

> StreamingToolExecutor fires tools mid-stream without waiting for the full response — the agent doesn't think, then act; it acts while still thinking.

## Overview

Every agent tutorial shows you the same diagram: model generates a response, response is parsed for tool calls, tools execute, results go back to the model. This is the textbook loop. It's correct. And it's not what production systems do.

Claude Code's agent loop, implemented in `query.ts`, breaks the "think then act" model in a fundamental way. Tools fire at the streaming delimiter — the moment the model emits a tool call boundary — not after the full response has arrived. The difference sounds minor. In practice it transforms the user experience from "waiting for long responses then watching tools execute" to "tools executing fluidly as the model reasons."

This chapter examines `query.ts` in detail: the `while(true)` loop structure, the `StreamingToolExecutor`, auto-compaction, and token budget management. These are the mechanisms that separate a weekend agent project from industrial-grade infrastructure.

## The while(true) Loop — Anatomy of query.ts

The agent loop is not a recursive function. It's not a chain of promises. It's a literal `while(true)` loop with explicit break conditions:

```typescript
// query.ts — simplified structure
async function* runQuery(
  messages: Message[],
  context: AgentContext,
): AsyncGenerator<AssistantMessage> {
  while (true) {
    // 1. Call the model (streaming)
    const stream = await anthropic.messages.stream({
      messages,
      system: context.systemPrompt,
      tools: context.tools,
    })

    // 2. Process stream with StreamingToolExecutor
    const { response, toolResults } = await streamingToolExecutor.process(stream)

    // 3. Yield assistant message
    yield response

    // 4. Break conditions
    if (response.stop_reason === 'end_turn') break
    if (response.stop_reason === 'max_tokens') break
    if (toolResults.length === 0) break

    // 5. Append tool results and continue
    messages = [...messages, response, ...toolResults]
  }
}
```

The loop continues as long as the model produces tool calls. When it produces a final `end_turn` with no tool calls, the loop exits. This is the fundamental mechanism behind multi-step agent behavior: the model can chain arbitrarily many tool calls before delivering a final answer.

Break conditions are critical safety valves. Without them, a model that keeps generating tool calls forever would loop indefinitely. The `max_tokens` break handles budget exhaustion, and checking `toolResults.length === 0` catches the edge case where `tool_use` stop reason arrives but no actual tool blocks were found.

## StreamingToolExecutor — Firing Mid-Stream

This is the key differentiator. The standard approach:

```typescript
// Naive: wait for full response, then parse tools
const response = await anthropic.messages.create({ ... }) // blocks until complete
const toolBlocks = response.content.filter(b => b.type === 'tool_use')
const toolResults = await Promise.all(toolBlocks.map(executeToolCall))
```

The production approach:

```typescript
// StreamingToolExecutor: fire tools at the delimiter
for await (const event of stream) {
  if (event.type === 'content_block_stop' && event.index === toolCallIndex) {
    // Tool call JSON is complete — fire it NOW, don't wait for end_turn
    const pendingResult = executeToolCall(currentToolBlock)
    pendingResults.push(pendingResult)
  }

  if (event.type === 'message_stop') {
    // All tool results should already be running or complete
    const toolResults = await Promise.all(pendingResults)
  }
}
```

The insight: a streaming response from the Anthropic API emits `content_block_stop` events for each content block as it completes. A tool call block is complete (parseable JSON) at `content_block_stop` — even though the overall message hasn't ended. The `StreamingToolExecutor` listens for these delimiters and fires tool execution immediately.

For responses with multiple tool calls, this means tool 1 can be executing while tool 2 is still being streamed from the model. The parallelism is real and measurable, especially for slow tools like file system operations or shell commands.

## Auto-Compact — Transparent Context Management

One of the most user-invisible features in Claude Code is what happens when you've had a long conversation and the context window fills up. The naive approach: throw an error or silently truncate. The production approach: transparently summarize and continue.

```typescript
// query.ts — context overflow handling
const tokenUsage = await estimateTokens(messages)

if (tokenUsage > CONTEXT_WINDOW * 0.95) {
  // Trigger auto-compact before the next API call
  const compactedMessages = await autoCompact(messages, context)
  messages = compactedMessages
  // User never sees this — conversation continues seamlessly
}
```

Auto-compact fires when context usage exceeds 95% of the window. It invokes the full compression pipeline (covered in Chapter 6) and replaces the message history with a summarized version that preserves the most important context while freeing space for new tokens.

The threshold of 95% is deliberate — it fires before the window is full, leaving headroom so the compaction summary itself doesn't overflow. This is a detail that matters: compact *before* you're stuck, not after.

## Token Budget Management — tokenBudget.ts

Token budgets are constraints, not suggestions. The `tokenBudget.ts` module tracks token expenditure across several dimensions:

```typescript
interface TokenBudget {
  // Hard limits
  maxOutputTokens: number      // Model's max output per request
  contextWindowSize: number    // Total context window (model-dependent)

  // Soft tracking
  inputTokensUsed: number      // Tokens consumed by messages + system prompt
  outputTokensUsed: number     // Tokens generated this session
  cacheHits: number            // Tokens served from prompt cache
  cacheMisses: number          // Tokens that had to be processed fresh
}
```

The budget affects several decisions in the loop:

1. **When to trigger auto-compact**: based on `inputTokensUsed / contextWindowSize`
2. **How to truncate tool results**: large results are snipped to stay within budget
3. **Whether to continue generating**: if `outputTokensUsed` approaches `maxOutputTokens`, force stop

Budget awareness throughout the loop prevents the most common failure mode of naive agents: hitting the context limit with no graceful recovery, causing the entire conversation to fail.

```
│         │           │

└─────────────────────┘
```

## Key Takeaways

- The agent loop is a `while(true)` with explicit break conditions — not recursion, not a promise chain.
- `StreamingToolExecutor` fires tool calls at `content_block_stop` events, not after `message_stop` — parallelism is built into the stream processing.
- Auto-compact triggers at 95% context usage and is entirely transparent to the user — conversations never hard-fail due to context overflow.
- Token budget tracking is proactive, not reactive — decisions are made before limits are hit, not after errors occur.
- For multiple concurrent tool calls in one response, streaming execution means later tools start executing before earlier ones have finished — genuine parallelism within a single model turn.
