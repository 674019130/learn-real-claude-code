# Observability

> Circuit breaker (max 3 consecutive failures) prevents infinite retry loops; 11-step exponential backoff handles transient errors — together they make Claude Code resilient without being fragile.

## Overview

A production agent has to deal with the world as it actually is: flaky APIs, transient network errors, rate limits, partial failures mid-stream. An agent that retries indefinitely is dangerous — it can drain an entire context window and burn through API credits on what turns out to be a permanent failure. An agent that fails on first error is fragile. The challenge is threading the needle: recover from the recoverable, stop quickly on the unrecoverable.

Claude Code approaches this with two overlapping systems: a detailed observability layer that tracks what is happening (StatsD metrics + Sentry errors + cost tracking), and a recovery pipeline that controls what happens when things go wrong (exponential backoff + circuit breaker). Together they make failure visible and bounded.

## Dual Telemetry Channels

Claude Code emits observability data on two separate channels with different purposes:

**StatsD** receives metrics: counters, gauges, and histograms that answer quantitative questions about system behavior. Request counts, latency distributions, token throughput, tool execution times — anything you would put on a dashboard.

**Sentry** receives errors: structured exception reports with stack traces, user context, breadcrumbs, and environment metadata. When something unexpected happens — an unhandled exception, an API response in an unexpected format, a tool that returns an error — Sentry captures enough context to diagnose the root cause.

```
┌──────────────────────────────────────────────────┐
│                  Claude Code Process              │
│                                                   │
│  [API call] → StatsD: increment('api.requests')  │
│             → StatsD: timing('api.latency', ms)  │
│                                                   │
│  [Exception] → Sentry: captureException(err)     │
│              → Sentry: breadcrumbs + context      │
└──────────────────────────────────────────────────┘
```

The two channels are deliberately separate. StatsD is fire-and-forget UDP — minimal latency impact, suitable for high-frequency events like per-token metrics. Sentry is structured and reliable — appropriate for error events where you need guaranteed delivery and rich context.

## Cost Tracking

`cost-tracker.ts` maintains a running account of token usage across the session:

```typescript
interface CostRecord {
  model: string;
  inputTokens: number;
  outputTokens: number;
  cacheReadTokens: number;
  cacheWriteTokens: number;
  costUsd: number;
  timestamp: number;
}

class CostTracker {
  private records: CostRecord[] = [];

  record(usage: TokenUsage, model: string): void {
    const cost = calculateCost(usage, model);
    this.records.push({ ...usage, model, costUsd: cost, timestamp: Date.now() });
    statsD.gauge('session.cost_usd', this.totalCost());
  }

  totalCost(): number {
    return this.records.reduce((sum, r) => sum + r.costUsd, 0);
  }
}
```

`costHook.ts` wires this into the API call lifecycle. After every response from the Anthropic API, the hook fires, extracts the usage metadata from the response headers, and records it. This means the cost tracker always reflects reality — not an estimate, but the actual usage reported by the API.

The tracker differentiates between standard input tokens, cached read tokens (much cheaper), and cache write tokens. This matters for understanding the real cost of prompt caching — knowing how much you're saving in cache reads versus paying for cache writes.

## 11-Step Exponential Backoff

The error recovery pipeline handles transient failures — rate limits, temporary service degradation, network hiccups. It uses exponential backoff with 11 defined steps:

```
Attempt 1:  immediate retry
Attempt 2:  wait  1s
Attempt 3:  wait  2s
Attempt 4:  wait  4s
Attempt 5:  wait  8s
Attempt 6:  wait 16s
Attempt 7:  wait 32s
Attempt 8:  wait 64s  (~1 min)
Attempt 9:  wait 128s (~2 min)
Attempt 10: wait 256s (~4 min)
Attempt 11: final attempt before giving up
```

The specific timing is calibrated against real-world API behavior: rate limit windows, service restart times, typical transient failure durations. Not every error gets all 11 retries — the recovery pipeline classifies errors first:

- **Retryable**: 429 (rate limit), 503 (service unavailable), network timeout → full backoff sequence
- **Non-retryable**: 400 (bad request), 401 (unauthorized), 404 (not found) → immediate failure, no retry
- **Ambiguous**: connection reset, partial response → limited retry (3 attempts, short backoff)

## Circuit Breaker

The backoff sequence handles the case where the API will eventually recover. The circuit breaker handles the case where it won't — or where repeated failures indicate something systemic.

After 3 consecutive failures, the circuit breaker opens:

```
State: CLOSED (normal operation)
  ↓ API call fails (1st consecutive)
State: CLOSED (failure count: 1)
  ↓ API call fails (2nd consecutive)
State: CLOSED (failure count: 2)
  ↓ API call fails (3rd consecutive)
State: OPEN  ← circuit breaks, stop retrying
  ↓ (after cooldown period)
State: HALF-OPEN (probe with single request)
  ↓ Success → State: CLOSED
  ↓ Failure → State: OPEN (reset timer)
```

The circuit breaker prevents the worst-case failure mode for an agent in a loop: a permanent API failure that triggers the full 11-step backoff, which completes and retries, which hits the full backoff again — an infinite loop consuming context window and time while producing nothing.

With the circuit breaker, 3 consecutive failures breaks the loop. The agent surfaces the error to the user rather than silently burning context on an operation that cannot succeed.

## Why Observability Comes Before Recovery

It might seem that recovery logic is more important than observability — after all, recovering from errors is what keeps the system running. But observability is what makes recovery trustworthy.

Without cost tracking, you don't know if the backoff sequence is burning too much budget. Without StatsD metrics, you can't tell if the circuit breaker is triggering too aggressively or not aggressively enough. Without Sentry errors, you can't distinguish "transient rate limit" from "the API is rejecting this specific request because of a bug in the prompt."

Observability is not instrumentation added after the fact. It is the foundation that makes every other reliability mechanism interpretable.

## Key Takeaways

- StatsD handles high-frequency metrics via fire-and-forget UDP; Sentry handles structured error capture with rich context — two channels for two different kinds of observability data
- `cost-tracker.ts` records actual token usage (including cache reads/writes) per API call via `costHook.ts`, maintaining real-time session cost
- 11-step exponential backoff handles transient errors with calibrated wait intervals; error classification determines which errors warrant retry
- Circuit breaker opens after 3 consecutive failures, preventing infinite retry loops that drain context window
- Observability is prerequisite to recovery: without knowing what is happening, you cannot know whether your recovery mechanisms are working
