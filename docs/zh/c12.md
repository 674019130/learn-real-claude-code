# 第十二章 -- 终端里的 React：自定义 Reconciler 与三个对象池

> CharPool 用 ASCII 快速路径（`Int32Array[128]`）绕过 Map 查找；StylePool 的 bit 0 编码"空格是否可见"，让渲染器用一个位掩码跳过无效格子。

---

## 概述

Claude Code 的终端界面不是 ncurses，不是手写 ANSI 转义——它是完整的 React 19 应用，跑在一个 fork 版 Ink 之上。核心由四个层组成：

1. **自定义 Reconciler**：`react-reconciler` 适配终端 DOM（`ink-box`、`ink-text`）。
2. **Yoga 布局**：Facebook 的 Flexbox 引擎，每个字符 = 一个"像素"。
3. **三个对象池**：`CharPool`、`StylePool`、`HyperlinkPool`，将 O(n) 字符串比较变为 O(1) 整数比较。
4. **双缓冲 + blit 优化**：前后 Screen 交换，`Output` 操作队列批处理写入和 blit，只发送 diff。

本章所有代码块均来自真实源码，标注文件名和行号。

---

## 1. 自定义 Reconciler：React 19 适配

`reconciler.ts` 调用 `react-reconciler` 的 `createReconciler`，为终端定义完整的 host config。注意类型签名中 `UpdatePayload` 是 `null`——React 19 在 `commitUpdate` 中直接传递新旧 props，不再需要 payload：

```typescript
// ink/reconciler.ts:224-239

  TextNode,
  DOMElement,
  unknown,
  unknown,
  DOMElement,
  HostContext,
  null, // UpdatePayload - not used in React 19
  NodeJS.Timeout,
  -1,
  null
>({
```

`createInstance` 负责将 JSX 元素变为终端 DOM 节点。`isInsideText` 检查防止 `<Box>` 被嵌套在 `<Text>` 内部——终端没有 inline-block，这种嵌套会破坏布局：

```typescript
// ink/reconciler.ts:331-358
createInstance(
  originalType: ElementNames,
  newProps: Props,
  _root: DOMElement,
  hostContext: HostContext,
  internalHandle?: unknown,
): DOMElement {
  if (hostContext.isInsideText && originalType === 'ink-box') {
    throw new Error(`<Box> can't be nested inside <Text> component`)
  }

  const type =
    originalType === 'ink-text' && hostContext.isInsideText
      ? 'ink-virtual-text'
      : originalType

  const node = createNode(type)
```

`commitUpdate` 直接做新旧 props 的 shallow diff，只对变化的属性调用 `setAttribute` 或 `applyStyles`：

```typescript
// ink/reconciler.ts:426-458
commitUpdate(
  node: DOMElement,
  _type: ElementNames,
  oldProps: Props,
  newProps: Props,
): void {
  const props = diff(oldProps, newProps)
  const style = diff(oldProps['style'] as Styles, newProps['style'] as Styles)

  if (props) {
    for (const [key, value] of Object.entries(props)) {
      if (key === 'style') {
        setStyle(node, value as Styles)
        continue
      }
      // ...
    }
  }

  if (style && node.yogaNode) {
    applyStyles(node.yogaNode, style, newProps['style'] as Styles)
  }
},
```

### 为什么不用浏览器 DOM 模拟？

终端"DOM"只有 7 种节点类型（`ink-root`、`ink-box`、`ink-text`、`ink-virtual-text`、`ink-link`、`ink-progress`、`ink-raw-ansi`）。浏览器 DOM 的 `document.createElement` 带着事件冒泡、CSS 级联、CSSOM 等巨大开销。自定义 DOM 的节点创建就是一个普通对象分配加一个 Yoga 节点：

```typescript
// ink/dom.ts:110-132
export const createNode = (nodeName: ElementNames): DOMElement => {
  const needsYogaNode =
    nodeName !== 'ink-virtual-text' &&
    nodeName !== 'ink-link' &&
    nodeName !== 'ink-progress'
  const node: DOMElement = {
    nodeName,
    style: {},
    attributes: {},
    childNodes: [],
    parentNode: undefined,
    yogaNode: needsYogaNode ? createLayoutNode() : undefined,
    dirty: false,
  }

  if (nodeName === 'ink-text') {
    node.yogaNode?.setMeasureFunc(measureTextNode.bind(null, node))
  }
  // ...
}
```

## 2. Yoga 布局与 dirty 标记

每次 DOM 变更（属性、子节点增删）都通过 `markDirty` 向上冒泡，标记整条祖先链为 dirty，同时对 text/raw-ansi 叶子节点调用 `yogaNode.markDirty()` 触发重新测量：

```typescript
// ink/dom.ts:393-413
export const markDirty = (node?: DOMNode): void => {
  let current: DOMNode | undefined = node
  let markedYoga = false

  while (current) {
    if (current.nodeName !== '#text') {
      ;(current as DOMElement).dirty = true
      if (
        !markedYoga &&
        (current.nodeName === 'ink-text' ||
          current.nodeName === 'ink-raw-ansi') &&
        current.yogaNode
      ) {
        current.yogaNode.markDirty()
        markedYoga = true
      }
    }
    current = current.parentNode
  }
}
```

`setStyle` 和 `setTextStyles` 内部做 shallow equal 检查——React 每次 render 都创建新的 style 对象，即使值没变。没有这个检查，每帧都会触发 Yoga 重新计算：

```typescript
// ink/dom.ts:266-274
export const setStyle = (node: DOMNode, style: Styles): void => {
  if (stylesEqual(node.style, style)) {
    return
  }
  node.style = style
  markDirty(node)
}
```

### 为什么不用 CSS Grid？

Yoga 只实现 Flexbox，不支持 Grid。但终端 UI 几乎不需要 Grid——内容是线性流式的（消息列表、工具输出），Flexbox 的 `flexDirection: 'column'` 天然匹配。Yoga 的 WASM 编译版在每帧布局计算上只需 1-5ms，用 CSS Grid 的通用引擎会慢一个数量级。

## 3. 三个对象池：CharPool、StylePool、HyperlinkPool

这是性能关键。双缓冲 diff 需要逐格比较 10,000+ 个格子（200x50 终端），对象池将字符串比较转为整数比较。

**CharPool** 为 ASCII 字符提供 `Int32Array` 快速路径，跳过 `Map.get` 的 hash 计算：

```typescript
// ink/screen.ts:21-53
export class CharPool {
  private strings: string[] = [' ', ''] // Index 0 = space, 1 = empty (spacer)
  private stringMap = new Map<string, number>([
    [' ', 0],
    ['', 1],
  ])
  private ascii: Int32Array = initCharAscii() // charCode -> index, -1 = not interned

  intern(char: string): number {
    // ASCII fast-path: direct array lookup instead of Map.get
    if (char.length === 1) {
      const code = char.charCodeAt(0)
      if (code < 128) {
        const cached = this.ascii[code]!
        if (cached !== -1) return cached
        const index = this.strings.length
        this.strings.push(char)
        this.ascii[code] = index
        return index
      }
    }
    const existing = this.stringMap.get(char)
    if (existing !== undefined) return existing
    // ...
  }
}
```

**StylePool** 更精巧——bit 0 编码"这个样式在空格字符上是否可见"（背景色、inverse、underline）。渲染器跳过无样式空格只需检查 `id & 1`，不需要解析 AnsiCode 数组：

```typescript
// ink/screen.ts:129-140
intern(styles: AnsiCode[]): number {
  const key = styles.length === 0 ? '' : styles.map(s => s.code).join('\0')
  let id = this.ids.get(key)
  if (id === undefined) {
    const rawId = this.styles.length
    this.styles.push(styles.length === 0 ? [] : styles)
    id =
      (rawId << 1) |
      (styles.length > 0 && hasVisibleSpaceEffect(styles) ? 1 : 0)
    this.ids.set(key, id)
  }
  return id
}
```

StylePool 还缓存样式间的转换序列。从 `(fromId, toId)` 到 ANSI 字符串只计算一次：

```typescript
// ink/screen.ts:153-163
transition(fromId: number, toId: number): string {
  if (fromId === toId) return ''
  const key = fromId * 0x100000 + toId
  let str = this.transitionCache.get(key)
  if (str === undefined) {
    str = ansiCodesToString(diffAnsiCodes(this.get(fromId), this.get(toId)))
    this.transitionCache.set(key, str)
  }
  return str
}
```

**HyperlinkPool** 结构最简单——index 0 = 无链接，其余按需 intern：

```typescript
// ink/screen.ts:57-75
export class HyperlinkPool {
  private strings: string[] = [''] // Index 0 = no hyperlink
  private stringMap = new Map<string, number>()

  intern(hyperlink: string | undefined): number {
    if (!hyperlink) return 0
    let id = this.stringMap.get(hyperlink)
    if (id === undefined) {
      id = this.strings.length
      this.strings.push(hyperlink)
      this.stringMap.set(hyperlink, id)
    }
    return id
  }
}
```

### 为什么不用 WeakMap 或 LRU 替代永生池？

三个池在整个会话期间不重置（HyperlinkPool 每 5 分钟重置一次除外）。字符和样式的组合是有限的——终端只有 128 个 ASCII 字符、几十种 SGR 组合。LRU 的淘汰逻辑反而增加每次 intern 的分支开销。永生池的 `Map.get` 是 O(1) 且无 GC 压力。

## 4. 双缓冲与 Output 操作队列

`ink.tsx` 持有 `frontFrame` 和 `backFrame` 两个 Frame。每帧渲染到 backFrame 的 Screen，然后与 frontFrame diff，最后交换：

```typescript
// ink/ink.tsx:99-100
private frontFrame: Frame;
private backFrame: Frame;
```

`Output` 类收集 `write`、`blit`、`clip`、`clear`、`shift` 操作，在 `get()` 中批处理写入 Screen buffer。`charCache` 跨帧复用——大多数行不变，tokenize + grapheme 聚类变为缓存命中：

```typescript
// ink/output.ts:196-205
reset(width: number, height: number, screen: Screen): void {
  this.width = width
  this.height = height
  this.screen = screen
  this.operations.length = 0
  resetScreen(screen, width, height)
  if (this.charCache.size > 16384) this.charCache.clear()
}
```

`blit` 操作从 prevScreen 整块拷贝未变化的区域，避免重新 tokenize 和 intern 那些没变的行：

```typescript
// ink/output.ts:209-211
blit(src: Screen, x: number, y: number, width: number, height: number): void {
  this.operations.push({ type: 'blit', src, x, y, width, height })
}
```

## 5. 渲染管道全貌

```
JSX 组件树
    | React 19 reconciler (ConcurrentRoot)
    v
ink DOM (ink-box, ink-text, ...)
    | dirty 标记向上冒泡
    v
Yoga calculateLayout (WASM)
    | 每个字符 = 1px
    v
renderNodeToOutput -> Output 操作队列
    | write/blit/clip/clear/shift
    v
Output.get() -> Screen (charId[], styleId[], hyperlinkId[])
    | 三个对象池 intern
    v
LogUpdate.diff(frontScreen, backScreen)
    | 逐格整数比较
    v
ANSI 转义序列 -> stdout
```

## 关键收获

- React 19 的自定义 Reconciler 让终端 UI 用标准 JSX 和 hooks 开发；`commitUpdate` 直接接收新旧 props，不再需要 `prepareUpdate` 返回 payload。
- Yoga Flexbox 引擎将每个终端字符视为 1px，用 `markDirty` 向上冒泡 + `setStyle` 的 shallow equal 检查避免不必要的布局重算。
- `CharPool` 用 `Int32Array[128]` 为 ASCII 字符提供数组直查快速路径，绕过 `Map.get` 的 hash 开销；`StylePool` 的 bit 0 编码空格可见性，让渲染器用位掩码跳过无效格子。
- `Output` 操作队列将 write/blit/clip/clear 批处理到 Screen buffer，`charCache` 跨帧复用避免重复 tokenize——稳态帧大部分格子走 blit 路径，渲染开销与变化面积成正比。
- 三个池的设计哲学：**终端字符/样式/链接的组合空间有限，永生池的 O(1) intern 优于 LRU 的淘汰开销**。
