# 110+ 字符串组装一条系统提示词

> 一个字面量字符串 `'__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__'` 把系统提示词切成两半。前半部分跨组织缓存，后半部分每会话重算。两个注册函数——一个安全、一个名字里带 `DANGEROUS_`——强制开发者在写每一个 section 时做出缓存决策。

## 问题

Claude Code 的系统提示词不是一个静态字符串。它是从超过 110 条独立字符串中动态组装出来的——身份定义、行为规则、工具描述、项目上下文、记忆文件、MCP 指令、环境信息。每条消息都要把这些发送给模型。

挑战在于成本：一条 50,000 token 的系统提示词，如果每次请求都重新处理，按 Opus 的输入价格算是 $0.75。一天 100 次请求就是 $75。但这 50,000 token 中大约 40,000 是不变的（身份、规则、工具描述）——如果能命中 Anthropic 的 prompt cache（90% 折扣），每次请求的成本降到 $0.21，每天省 $54。

问题是：如何把「不变的部分」和「每次都变的部分」干净地分开，同时允许 20 个 section 函数独立地决定自己属于哪一类？

## 为什么不用简单的字符串拼接

**被拒绝的方案：手写一个大字符串。** 系统提示词的内容来自 10+ 个模块，每个模块负责自己领域的指令。把所有内容写在一个文件里意味着每次修改工具描述都要编辑一个 3000 行的字符串。

**被拒绝的方案：基于内容哈希的缓存。** 可以对每个 section 做哈希，变了才重新发送。但 Anthropic 的 prompt cache 是基于前缀匹配的——只有当请求的系统提示词与缓存条目的前 N 个 token 完全一致时才命中。内容哈希无法利用这个机制。必须保证前缀稳定。

**被拒绝的方案：所有 section 每 turn 重算。** 简单、正确、但贵。大多数 section 在一次对话中根本不变（身份定义不会变、工具描述不会变）。只有 MCP 指令这种东西才会在 turn 之间改变——用户可能在另一个终端连接了新的 MCP 服务器。

## 两个注册函数——核心设计决策

整个系统的关键在于 `systemPromptSections.ts` 中的两个函数。每一个 section 都必须通过其中之一注册：

```typescript
// constants/systemPromptSections.ts —— 完整源码（69 行）

type ComputeFn = () => string | null | Promise<string | null>

type SystemPromptSection = {
  name: string
  compute: ComputeFn
  cacheBreak: boolean
}

/**
 * 创建一个被记忆化的 section。
 * 计算一次，缓存到 /clear 或 /compact。
 */
export function systemPromptSection(
  name: string,
  compute: ComputeFn,
): SystemPromptSection {
  return { name, compute, cacheBreak: false }
}

/**
 * 创建一个易变的 section，每个 turn 都重新计算。
 * 当值变化时会打断 prompt cache。
 * 要求给出为什么必须打断缓存的原因。
 */
export function DANGEROUS_uncachedSystemPromptSection(
  name: string,
  compute: ComputeFn,
  _reason: string,
): SystemPromptSection {
  return { name, compute, cacheBreak: true }
}
```

`DANGEROUS_` 前缀不是玩笑。它是一种 API 设计上的刻意摩擦——当你在代码里写下 `DANGEROUS_uncachedSystemPromptSection` 时，你被迫停下来想一想这个 section 是否真的需要每 turn 重算。`_reason` 参数是强制的：你必须用字符串解释为什么。它不会被日志记录或显示给用户——纯粹是调用点的文档：

```typescript
// 真实用例：MCP 服务器在 turn 之间可以连接或断开
DANGEROUS_uncachedSystemPromptSection(
  'mcp_instructions',
  () => isMcpInstructionsDeltaEnabled() ? null : getMcpInstructionsSection(),
  'MCP servers connect/disconnect between turns',  // ← 强制的原因
)
```

## 为什么不用一个 `volatile: boolean` 标志

**被拒绝的方案：`systemPromptSection(name, compute, { volatile: true })`。** 一个布尔标志太容易随手写 `true`。函数名里嵌入 `DANGEROUS_` 这个词让「选择打断缓存」变成一个需要主动决策的行为，而不是一个可以默认选中的选项。代码审查时，所有 `DANGEROUS_` 调用都是肉眼可见的警告信号。

**被拒绝的方案：自动检测变化。** 可以在每个 turn 比较 section 的输出，变了才标记为 dirty。但这意味着每个 turn 都要执行所有 compute 函数——即使结果没变，I/O 成本（读取 CLAUDE.md、查询 MCP 状态）已经发生了。显式的 `cacheBreak` 标志让不需要重算的 section 完全跳过 compute。

## Section 解析——并行计算、按名称缓存

组装系统提示词时，所有 section 通过 `Promise.all` 并行解析：

```typescript
// constants/systemPromptSections.ts (lines 43-58) —— 完整源码
export async function resolveSystemPromptSections(
  sections: SystemPromptSection[],
): Promise<(string | null)[]> {
  const cache = getSystemPromptSectionCache()

  return Promise.all(
    sections.map(async s => {
      if (!s.cacheBreak && cache.has(s.name)) {
        return cache.get(s.name) ?? null  // 缓存命中，跳过 compute
      }
      const value = await s.compute()
      setSystemPromptSectionCacheEntry(s.name, value)
      return value
    }),
  )
}
```

三个关键细节：

1. **并行解析。** `Promise.all` 意味着加载 memory 不会阻塞查询环境信息，环境信息不会阻塞 MCP 指令。各个 compute 函数相互独立。

2. **按名称缓存，不按内容。** 名为 `'memory'` 的 section 会一直返回缓存值直到缓存被清除。这是会话级的记忆化，不是内容寻址存储。

3. **`cacheBreak: true` 完全跳过缓存查找。** 每次都调用 `compute()`，但仍然会把结果存入缓存（供调试或其他潜在使用）。

## 缓存失效——只有两个触发点

```typescript
// constants/systemPromptSections.ts (lines 65-68) —— 完整源码
export function clearSystemPromptSections(): void {
  clearSystemPromptSectionState()   // 清除所有 section 缓存
  clearBetaHeaderLatches()          // 重置 beta header 锁存器
}
```

这个函数只在两个地方被调用：`/clear` 和 `/compact`。

这意味着：如果你在对话中途编辑了项目的 CLAUDE.md 文件，模型不会看到更新——直到运行 `/clear` 或 `/compact`。这是有意的取舍：在每个 turn 都重新读取所有 CLAUDE.md 文件会增加磁盘 I/O，而 CLAUDE.md 在一次对话中很少改变。

## 六个静态 Section——身份核心

边界之前的 section 构成模型的身份。它们在每个会话、每个用户、每个项目中完全相同：

```
Section 名称           来源函数                        内容
──────────────────────────────────────────────────────────────
intro                  getSimpleIntroSection()         "You are Claude Code..."
                                                       核心身份定义

system                 getSimpleSystemSection()        工具执行规则、
                                                       输出格式、hooks 行为

doing_tasks            getSimpleDoingTasksSection()    代码风格指导、
                                                       任务完成模式

actions                getActionsSection()             可逆性评估规则、
                                                       风险分级标准

using_tools            getUsingYourToolsSection()      何时用哪个工具、
                                                       工具选择启发式

tone_and_style         getSimpleToneAndStyleSection()  不用 emoji、简洁直接、
                                                       引用格式规范
```

这六个 section 编译进二进制文件，不依赖任何运行时状态。它们的缓存 scope 是 `'global'`——跨组织共享。一个静态内容的哈希值决定缓存命中。修改了 intro section 里的一个字？所有用户的缓存全部失效，直到新版本预热完成。

## 动态边界

边界本身是一个常量字符串：

```typescript
// constants/prompts.ts
const SYSTEM_PROMPT_DYNAMIC_BOUNDARY = '__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__'
```

它作为数组的一个元素出现在静态 section 和动态 section 之间。API 层用这个标记判断可缓存前缀在哪里结束。

```
┌─────────────────────────────────────────────────────┐
│  intro          "You are Claude Code..."             │
│  system         工具规则、输出格式                     │
│  doing_tasks    代码风格、任务指导                     │  scope: 'global'
│  actions        风险评估规则                           │  （跨组织缓存）
│  using_tools    工具选择启发式                         │
│  tone_and_style 风格规范                              │
│                                                     │
├── '__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__' ──────────────┤
│                                                     │
│  session_guidance    当前工具 + 技能列表               │
│  memory              CLAUDE.md 内容                   │
│  mcp_instructions ⚡ MCP 服务器指令                   │  scope: 每会话
│  claude_md           项目 CLAUDE.md                   │  （⚡ 标记的 section
│  environment         OS、shell、cwd、git              │    每 turn 重算，
│  deferred_tools      仅名称的工具列表                  │    其余按会话缓存）
│  ... (12-15 个 section)                               │
└─────────────────────────────────────────────────────┘
```

## 动态 Section——哪些安全、哪些危险

边界以下的 section 分两类：大多数用安全函数注册（一次对话内不变），少数用 `DANGEROUS_` 注册（每 turn 可能变）：

```typescript
// constants/prompts.ts (简化) —— 动态 section 注册

// 安全：工具列表和技能在会话启动时确定，此后不变
systemPromptSection('session_guidance', () =>
  getSessionSpecificGuidanceSection(enabledTools, skillToolCommands),
)

// 安全：memory 在会话启动时加载，/clear 前不变
systemPromptSection('memory', () => loadMemoryPrompt())

// 危险：MCP 服务器可以在任意两条消息之间连接或断开
DANGEROUS_uncachedSystemPromptSection(
  'mcp_instructions',
  () => isMcpInstructionsDeltaEnabled() ? null : getMcpInstructionsSection(),
  'MCP servers connect/disconnect between turns',
)
```

`session_guidance` 为什么是安全的？因为可用工具列表和技能命令在会话启动时确定，此后不会改变。如果用户安装了新的 MCP 服务器，工具列表会变——但那是 `mcp_instructions` 的职责，不是 `session_guidance` 的。

## 延迟工具——控制提示词在 200K token 以内

40+ 个工具，每个携带完整的 JSON schema，光工具描述就能消耗 50,000+ token。Claude Code 的方案是把工具分成两类：

```
完整注册（在系统提示词中携带完整 schema）：
  Read, Write, Edit, Bash, Glob, Grep, Skill —— 7 个核心工具

延迟注册（只有名称，没有 schema）：
  NotebookEdit, WebFetch, WebSearch, CronCreate, ... —— 18+ 个工具
```

当模型需要一个延迟工具时，它通过 `ToolSearchTool` 按需加载完整 schema：

```
模型想用 WebSearch：
  1. 在系统提示词的延迟工具列表中看到 "WebSearch"（仅名称）
  2. 调用 ToolSearchTool({ query: "select:WebSearch" })
  3. 收到完整的 JSON schema 定义
  4. 现在可以用正确的参数调用 WebSearch
```

这是一个两步调用模式——用一次额外的 API 往返换取系统提示词少 ~30,000 token。按 token 价格算，这笔交易非常划算。

## 为什么不用动态工具列表

**被拒绝的方案：只注册当前任务需要的工具。** 这需要预测模型会用哪些工具——在模型开始思考之前。预测不准意味着模型无法使用它需要的工具。全部注册更安全，但 token 成本太高。延迟加载是折中——模型知道所有工具的存在（看到名称），但只在需要时才付出 schema 的 token 成本。

**被拒绝的方案：把工具 schema 放在对话历史而非系统提示词中。** 系统提示词享受 prompt cache 折扣；对话历史不享受。7 个核心工具的 schema 放在静态 section 里，跨请求缓存，成本接近于零。

## KAIROS 模式——一套完全不同的提示词

当 Claude Code 在 proactive/KAIROS 模式下运行时，整个组装管线被替换。普通模式的 20 个 section 函数被一条精简的自主 agent 身份替代：

```
普通模式：
  20 个 section → 110+ 字符串 → "你是一个帮助开发者的助手"
  工具：全套 40+
  行为：等待指令、请求确认

KAIROS 模式：
  精简的 section → 最小化字符串 → "你是一个追求目标的自主 agent"
  工具：受限子集
  行为：主动行动、不等指令
```

这不是改了几个 section 的标志位。它是一条完全独立的代码路径，产出的是根本不同的 agent 身份。

## 成本计算

```
Prompt cache 定价（Anthropic Opus）：
  未缓存输入：    $15.00 / 1M tokens
  缓存命中：       $1.50 / 1M tokens  （90% 折扣）

典型系统提示词：
  静态 section：  ~40,000 tokens
  动态 section：  ~10,000 tokens

每请求成本（无缓存）：
  50,000 × $15.00/1M = $0.75

每请求成本（静态部分缓存命中）：
  40,000 × $1.50/1M  = $0.06  （缓存）
  10,000 × $15.00/1M = $0.15  （重算）
  总计：               $0.21

每请求节省：$0.54
每天 100 请求：节省 $54
```

那个 `'__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__'` 字符串不是一个装饰性分隔符。它是一个每天价值 $54 的架构决策。

## 组装管线

最终组装的结构：

```typescript
// constants/prompts.ts (简化)
async function assembleSystemPrompt(session: Session): Promise<(string | null)[]> {
  const staticSections = [
    systemPromptSection('intro', getSimpleIntroSection),
    systemPromptSection('system', getSimpleSystemSection),
    systemPromptSection('doing_tasks', getSimpleDoingTasksSection),
    systemPromptSection('actions', getActionsSection),
    systemPromptSection('using_tools', getUsingYourToolsSection),
    systemPromptSection('tone_and_style', getSimpleToneAndStyleSection),
  ]

  const dynamicSections = [
    systemPromptSection('session_guidance', ...),
    systemPromptSection('memory', ...),
    DANGEROUS_uncachedSystemPromptSection('mcp_instructions', ..., '...'),
    // ... 12-15 个更多 section
  ]

  const resolvedStatic = await resolveSystemPromptSections(staticSections)
  const resolvedDynamic = await resolveSystemPromptSections(dynamicSections)

  return [
    ...resolvedStatic,
    SYSTEM_PROMPT_DYNAMIC_BOUNDARY,
    ...resolvedDynamic,
  ].filter(s => s !== null)
}
```

返回值是字符串数组——不是拼接好的单一字符串。Anthropic 的 API 接受系统提示词作为 content block 数组，保持分离才能保留缓存边界的语义。

末尾的 `.filter(s => s !== null)` 是条件性 section 的工作方式——`compute()` 返回 `null` 的 section 被静默省略。没有 MCP 服务器连接？`getMcpInstructionsSection()` 返回 `null`，MCP 区块不出现在提示词中。

## 核心要点

- 系统提示词是由 ~20 个 section 函数产出的 110+ 条字符串的数组，被 `'__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__'` 切成全局缓存的前缀和每会话变动的后缀。
- 两个注册函数实施缓存纪律：`systemPromptSection()` 计算一次、缓存整个会话；`DANGEROUS_uncachedSystemPromptSection()` 每 turn 重算。函数名里的 `DANGEROUS_` 和强制的 `_reason` 参数是刻意的 API 摩擦，防止开发者随意打断缓存。
- Section 缓存只在 `/clear` 和 `/compact` 时清除。对话中途编辑 CLAUDE.md 不会被立即感知——这是有意的取舍，避免每 turn 都做磁盘 I/O。
- 延迟工具加载用一次额外 API 往返换取系统提示词少 ~30,000 token。7 个核心工具携带完整 schema，18+ 个工具只有名称，通过 `ToolSearchTool` 按需加载。
- 边界标记是每天价值 $54 的架构决策——40,000 token 以 90% 折扣缓存，每请求节省 $0.54。
