# 110+ 字符串拼成一个系统提示词

> SYSTEM_PROMPT_DYNAMIC_BOUNDARY 将静态（全局缓存）与动态（每会话）部分分开。

## 概述

Claude Code 的系统提示词不是一个静态字符串——它是在每次会话初始化时，从 10 多个来源动态组装的。项目的 `CLAUDE.md`、用户主目录的 `CLAUDE.md`、活跃文件上下文、工具描述、环境信息、持久化记忆……这些片段全部合并成一个发送给模型的系统提示词。

但动态组装有一个问题：每次请求的系统提示词都略有不同，Anthropic 的 Prompt Cache（提示词缓存）就无法命中，导致每次请求都付出完整的 token 处理成本。`SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 就是为了解决这个问题而存在的。

## context.ts：提示词的组装流程

提示词组装发生在 `context.ts`：

```typescript
// context.ts（概念性重现）
async function buildSystemPrompt(session: Session): Promise<string> {
  const parts: string[] = [];

  // === 静态部分（在 DYNAMIC_BOUNDARY 之上）===
  parts.push(BASE_SYSTEM_PROMPT);          // 核心指令，几乎不变
  parts.push(await loadUserClaude());       // ~/.claude/CLAUDE.md
  parts.push(buildToolDescriptions());     // 工具描述（启动时固定）
  parts.push(CAPABILITY_DECLARATIONS);     // 能力声明

  // === 分界线 ===
  parts.push(SYSTEM_PROMPT_DYNAMIC_BOUNDARY);

  // === 动态部分（每会话不同）===
  parts.push(await loadProjectClaude());   // 项目目录的 CLAUDE.md
  parts.push(buildEnvironmentInfo());      // 当前时间、OS、工作目录等
  parts.push(buildActiveFileContext());    // 当前打开的文件
  parts.push(await loadMemory());          // 持久化记忆（会随时间变化）

  return parts.filter(Boolean).join('\n\n');
}
```

`SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 是一个特殊字符串，标记静态内容和动态内容的边界。

## Prompt Cache：为什么边界很重要

Anthropic API 支持提示词缓存：如果请求的前 N 个 token 与之前的某个请求完全相同，服务器不会重新处理这些 token，而是直接从缓存返回——费用也大幅降低。

```
没有边界时（全部被污染）：
请求 1: [基础提示词 2000 token][环境信息：时间=09:00]
请求 2: [基础提示词 2000 token][环境信息：时间=09:01]
                                ↑ 时间不同，从这里开始缓存不命中
→ 每次请求都重新处理全部 2000 token

有边界时（静态部分稳定）：
请求 1: [基础提示词 2000 token][BOUNDARY][时间=09:00]
请求 2: [基础提示词 2000 token][BOUNDARY][时间=09:01]
         ←── 这 2000 token 从缓存命中 ──→ ← 只重新处理这部分
→ 每次请求只处理动态部分的 token
```

在实践中，静态部分（基础系统提示词 + 用户 CLAUDE.md + 工具描述）可能有 3000-5000 token，而每次都在变化的动态部分可能只有 200-500 token。通过边界分割，cache 命中率可以非常高，大幅降低 API 成本和延迟。

## CLAUDE.md 的三个层级

`CLAUDE.md` 文件以层级结构加载，全部合并到系统提示词中：

```
层级 1: ~/CLAUDE.md（用户全局）
  → 用户的个人偏好、常用工具、编码风格

层级 2: /path/to/project/CLAUDE.md（项目级）
  → 项目架构、技术栈、贡献规范

层级 3: /path/to/project/subdir/CLAUDE.md（子目录级）
  → 特定模块的规范（如果存在）
```

```typescript
// CLAUDE.md 层级加载（概念性重现）
async function loadProjectClaude(): Promise<string> {
  const locations = [
    os.homedir() + '/CLAUDE.md',
    ...walkUpDirectories(process.cwd()).map(dir => dir + '/CLAUDE.md'),
  ];

  const contents = await Promise.all(
    locations.map(loc => readFileSafe(loc)) // 不存在时返回空字符串
  );

  // 从最宽泛（全局）到最具体（当前目录）排列
  return contents
    .filter(Boolean)
    .join('\n\n---\n\n');
}
```

这种层级合并意味着：你在用户级 CLAUDE.md 写的"我偏好函数式风格"会影响所有项目，而项目级 CLAUDE.md 中的"这个项目用 PostgreSQL"只影响当前项目。具体规则覆盖通用规则，因为它排列在后面，在提示词中距离用户消息更近。

## 工具描述的生成

工具描述是系统提示词的重要组成部分，由 `tools.ts` 中的工具 schema 动态生成：

```typescript
function buildToolDescriptions(tools: Tool[]): string {
  // 工具的 Zod schema 被转换为 JSON Schema
  // 这部分在会话开始时固定，放在 BOUNDARY 之前
  return tools
    .map(tool => JSON.stringify(zodToJsonSchema(tool.schema)))
    .join('\n');
}
```

工具描述放在静态部分，因为工具集在会话期间不会改变。

## 环境信息：每次请求都不同

动态部分包含当前的环境快照：

```typescript
function buildEnvironmentInfo(): string {
  return [
    `当前时间：${new Date().toISOString()}`,
    `操作系统：${os.platform()} ${os.release()}`,
    `工作目录：${process.cwd()}`,
    `Shell：${process.env.SHELL}`,
    `Node.js 版本：${process.version}`,
  ].join('\n');
}
```

这类信息每次都不同（至少时间戳不同），所以放在 BOUNDARY 之后，不会污染前面的缓存命中。

## 关键收获

- 系统提示词从 10+ 个来源组装，包括多级 `CLAUDE.md`、工具描述、环境信息和记忆
- `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 将稳定内容（静态）与每次变化的内容（动态）分开
- 静态部分放在 BOUNDARY 之前可以命中 Anthropic Prompt Cache，大幅降低 API 成本
- `CLAUDE.md` 按全局/项目/子目录三级层次合并，具体规则覆盖通用规则
- 工具描述在会话开始时固定，属于静态部分；时间戳、工作目录等属于动态部分
