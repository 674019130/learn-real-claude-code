# MCP：无限扩展协议

> MCP 工具通过 `fetchToolsForClient` 桥接为原生 Tool 接口，ToolSearchTool 使用加权关键词评分实现延迟加载，自动阈值在 MCP 工具占上下文 10% 时触发。

## 概述

MCP（Model Context Protocol）是 Claude Code 连接外部工具的标准协议。但"标准协议"四个字掩盖了源码中的工程复杂度：5 种传输类型、一个完整的工具桥接层、一个带加权评分的搜索算法、以及一套三模式的延迟加载决策系统。

真实的 MCP 实现不是"连接服务器、调用工具"那么简单。

## 工具桥接：fetchToolsForClient

MCP 协议返回的工具定义需要转换为 Claude Code 的内部 `Tool` 接口。这个桥接发生在 `fetchToolsForClient` 中：

```typescript
// services/mcp/client.ts:1743
export const fetchToolsForClient = memoizeWithLRU(
  async (client: MCPServerConnection): Promise<Tool[]> => {
    if (client.type !== 'connected') return []

    const result = (await client.client.request(
      { method: 'tools/list' },
      ListToolsResultSchema,
    )) as ListToolsResult

    const toolsToProcess = recursivelySanitizeUnicode(result.tools)

    return toolsToProcess.map((tool): Tool => {
      const fullyQualifiedName = buildMcpToolName(client.name, tool.name)
      return {
        ...MCPTool,    // 继承 MCPTool 的基础实现
        name: skipPrefix ? tool.name : fullyQualifiedName,
        mcpInfo: { serverName: client.name, toolName: tool.name },
        isMcp: true,
        searchHint:
          typeof tool._meta?.['anthropic/searchHint'] === 'string'
            ? tool._meta['anthropic/searchHint'].replace(/\s+/g, ' ').trim()
            : undefined,
        alwaysLoad: tool._meta?.['anthropic/alwaysLoad'] === true,
        async prompt() {
          const desc = tool.description ?? ''
          return desc.length > MAX_MCP_DESCRIPTION_LENGTH
            ? desc.slice(0, MAX_MCP_DESCRIPTION_LENGTH) + '... [truncated]'
            : desc
        },
        isConcurrencySafe() {
          return tool.annotations?.readOnlyHint ?? false
        },
        isDestructive() {
          return tool.annotations?.destructiveHint ?? false
        },
        inputJSONSchema: tool.inputSchema as Tool['inputJSONSchema'],
        // ...
      }
    })
  },
)
```

关键细节：

1. **名称规范化**：`buildMcpToolName` 生成 `mcp__{server}__{tool}` 格式的全限定名。
2. **描述截断**：`MAX_MCP_DESCRIPTION_LENGTH = 2048` -- OpenAPI 生成的 MCP 服务器会输出 15-60KB 的描述，必须截断。
3. **MCP annotations 映射**：`readOnlyHint` 映射到 `isConcurrencySafe()`，`destructiveHint` 映射到 `isDestructive()` -- MCP 协议的标注直接影响权限和并发决策。
4. **searchHint**：MCP 服务器可以通过 `_meta['anthropic/searchHint']` 提供搜索关键词，提升 ToolSearchTool 的匹配准确性。
5. **alwaysLoad**：`_meta['anthropic/alwaysLoad']` 让工具绕过延迟加载，始终出现在初始 prompt 中。

## MCPTool：可覆盖的骨架

所有 MCP 工具共享一个基础模板 `MCPTool`，桥接时覆盖关键方法：

```typescript

export const MCPTool = buildTool({

  get inputSchema(): InputSchema {

  },
```

`inputSchema` 是 `z.object({}).passthrough()` -- 接受任意对象。MCP 工具的真实 schema 在 `inputJSONSchema` 字段中，由 API 层直接传递给模型。

## ToolSearchTool：加权关键词搜索

ToolSearchTool 不是简单的字符串匹配。它有两种搜索模式和一套加权评分系统。

### 模式 1：select 直接选取

```typescript
// tools/ToolSearchTool/ToolSearchTool.ts:363
const selectMatch = query.match(/^select:(.+)$/i)
if (selectMatch) {
  const requested = selectMatch[1]!
    .split(',').map(s => s.trim()).filter(Boolean)

  const found: string[] = []
  for (const toolName of requested) {
    const tool =
      findToolByName(deferredTools, toolName) ??
      findToolByName(tools, toolName)  // 已加载的也能 select
    if (tool) {
      if (!found.includes(tool.name)) found.push(tool.name)
    }
  }
  return buildSearchResult(found, query, deferredTools.length)
}
```

`select:Read,Edit,Grep` -- 按名称直接获取工具定义。注意它也搜索已加载的工具列表 -- 选择一个已加载工具是无害的 no-op，避免了模型的重试循环。

### 模式 2：关键词搜索

```typescript
// tools/ToolSearchTool/ToolSearchTool.ts:186
async function searchToolsWithKeywords(
  query: string, deferredTools: Tools, tools: Tools, maxResults: number,
): Promise<string[]> {
  // Fast path: exact match
  const exactMatch =
    deferredTools.find(t => t.name.toLowerCase() === queryLower) ??
    tools.find(t => t.name.toLowerCase() === queryLower)
  if (exactMatch) return [exactMatch.name]

  // MCP prefix match
  if (queryLower.startsWith('mcp__') && queryLower.length > 5) {
    const prefixMatches = deferredTools
      .filter(t => t.name.toLowerCase().startsWith(queryLower))
    if (prefixMatches.length > 0) return prefixMatches.map(t => t.name)
  }

  // Weighted scoring
  const scored = await Promise.all(
    candidateTools.map(async tool => {
      const parsed = parseToolName(tool.name)
      const description = await getToolDescriptionMemoized(tool.name, tools)
      let score = 0
      for (const term of allScoringTerms) {
        const pattern = termPatterns.get(term)!
        if (parsed.parts.includes(term))
          score += parsed.isMcp ? 12 : 10       // 精确部分匹配
        else if (parsed.parts.some(part => part.includes(term)))
          score += parsed.isMcp ? 6 : 5         // 部分子串匹配
        if (hintNormalized && pattern.test(hintNormalized))
          score += 4                             // searchHint 匹配
        if (pattern.test(descNormalized))
          score += 2                             // 描述匹配
      }
      return { name: tool.name, score }
    }),
  )

  return scored.filter(i => i.score > 0)
    .sort((a, b) => b.score - a.score)
    .slice(0, maxResults).map(i => i.name)
}
```

评分权重：

| 匹配类型 | MCP 工具 | 普通工具 |
|---------|---------|---------|
| 名称部分精确匹配 | 12 | 10 |
| 名称子串匹配 | 6 | 5 |
| searchHint 词边界匹配 | 4 | 4 |
| 描述词边界匹配 | 2 | 2 |

MCP 工具的名称匹配权重更高，因为 `mcp__server__action` 格式的 server name 部分是最强的意图信号。

## 返回格式：tool_reference

ToolSearchTool 的返回不是 JSON 文本，而是 `tool_reference` content block -- API 层面的一等公民：

```typescript
// tools/ToolSearchTool/ToolSearchTool.ts:444
mapToolResultToToolResultBlockParam(content: Output, toolUseID: string) {
  if (content.matches.length === 0) {
    return { type: 'tool_result', tool_use_id: toolUseID,
      content: 'No matching deferred tools found' }
  }
  return {
    type: 'tool_result', tool_use_id: toolUseID,
    content: content.matches.map(name => ({
      type: 'tool_reference' as const,
      tool_name: name,
    })),
  } as unknown as ToolResultBlockParam
}
```

`tool_reference` 告诉 API："将这个工具的完整 schema 注入到模型的上下文中"。这不是一个 JSON 字符串让模型解析 -- 是 API 服务端执行的引用展开。

## 延迟加载决策：三模式系统

延迟加载不是简单的开/关，而是三种模式：

```typescript
// utils/toolSearch.ts:161
export type ToolSearchMode = 'tst' | 'tst-auto' | 'standard'

// utils/toolSearch.ts:172
export function getToolSearchMode(): ToolSearchMode {
  if (isEnvTruthy(process.env.CLAUDE_CODE_DISABLE_EXPERIMENTAL_BETAS))
    return 'standard'

  const value = process.env.ENABLE_TOOL_SEARCH
  if (isAutoToolSearchMode(value)) return 'tst-auto'
  if (isEnvTruthy(value)) return 'tst'
  if (isEnvDefinedFalsy(process.env.ENABLE_TOOL_SEARCH)) return 'standard'
  return 'tst'  // default: always defer
}
```

- **tst**：始终延迟加载（默认行为）。
- **tst-auto**：当 MCP 工具描述超过上下文窗口的 N% 时自动启用。
- **standard**：不延迟，所有工具直接加载。

`tst-auto` 的阈值计算：

```typescript
// utils/toolSearch.ts:49
const DEFAULT_AUTO_TOOL_SEARCH_PERCENTAGE = 10 // 10%

// utils/toolSearch.ts:104
function getAutoToolSearchTokenThreshold(model: string): number {
  const betas = getMergedBetas(model)
  const contextWindow = getContextWindowForModel(model, betas)
  const percentage = getAutoToolSearchPercentage() / 100
  return Math.floor(contextWindow * percentage)
}
```

默认 10% -- 如果模型有 200K token 上下文窗口，阈值是 20K token。超过这个数就自动启用延迟加载。

## 哪些工具永远不会被延迟

`isDeferredTool` 函数中有一系列永不延迟的例外：

```typescript
// tools/ToolSearchTool/prompt.ts:62
export function isDeferredTool(tool: Tool): boolean {
  if (tool.alwaysLoad === true) return false         // MCP _meta opt-out
  if (tool.isMcp === true) return true               // MCP 工具始终延迟
  if (tool.name === TOOL_SEARCH_TOOL_NAME) return false  // 自己不能延迟
  if (feature('FORK_SUBAGENT') && tool.name === AGENT_TOOL_NAME) {
    if (m.isForkSubagentEnabled()) return false      // fork 模式下 Agent 不延迟
  }
  return tool.shouldDefer === true
}
```

ToolSearchTool 永远不延迟（否则谁来发现它？），fork 模式下 AgentTool 不延迟（需要第一轮就能 fork），MCP 工具可以通过 `_meta['anthropic/alwaysLoad']` 自行 opt-out。

## Compaction 时的工具发现保护

延迟加载引入了一个难题：compaction 删除旧消息时，`tool_reference` 也会被删除。源码用两种策略保护：

```typescript
// utils/toolSearch.ts:545
export function extractDiscoveredToolNames(messages: Message[]): Set<string> {
  const discoveredTools = new Set<string>()

  for (const msg of messages) {
    // Compact boundary carries the pre-compact discovered set
    if (msg.type === 'system' && msg.subtype === 'compact_boundary') {
      const carried = msg.compactMetadata?.preCompactDiscoveredTools
      if (carried) {
        for (const name of carried) discoveredTools.add(name)
      }
      continue
    }
    // Scan user messages for tool_reference blocks
    if (msg.type !== 'user') continue
    for (const block of content) {
      if (isToolResultBlockWithContent(block)) {
        for (const item of block.content) {
          if (isToolReferenceWithName(item))
            discoveredTools.add(item.tool_name)
        }
      }
    }
  }
  return discoveredTools
}
```

Compaction 前将已发现的工具名快照到 `compactMetadata.preCompactDiscoveredTools`，恢复后从 compact boundary 消息中读回。

## 连接管理：批量并发 + 指数退避

MCP 服务器连接不是逐个建立的，而是分批：

## 为什么不用 GraphQL/REST 统一 API？

一个统一的 API gateway 看似更简洁，但 MCP 选择了 per-server 独立连接：

1. **传输异构性**：stdio 子进程、SSE 流、WebSocket 双向通道、in-process SDK -- 这些传输语义根本不同，统一 API 会抹平每种传输的优势。
2. **生命周期独立**：一个 MCP 服务器崩溃不应该影响其他服务器。独立连接意味着独立的重连和错误处理。
3. **认证隔离**：不同服务器可能需要不同的 OAuth 流程、API key、或者根本不需要认证（如 IDE 服务器），统一 gateway 的认证模型会变成最大公分母。

## 为什么不把所有工具预加载？

理论上 200K 上下文窗口够大，为什么不直接加载？

1. **Token 经济学**：10% 阈值（默认 20K token）不是小数字 -- 每次 API 调用都重复发送这些 token，即使 cache hit，cost 和 latency 仍然线性增长。
2. **工具间干扰**：大量工具描述在 prompt 中竞争模型注意力。实验表明工具超过 40 个时，模型选择正确工具的准确率下降。
3. **Haiku 不支持**：`tool_reference` 是 beta API 特性，Haiku 模型不支持，延迟加载在 Haiku 上会自动降级为 standard 模式。

## 关键收获

- MCP 工具通过 `fetchToolsForClient` 桥接为原生 `Tool` 接口，包括名称规范化（`mcp__{server}__{tool}`）、描述截断（2048 字符上限）、MCP annotations 到权限/并发属性的映射。
- ToolSearchTool 有两种模式（select 直接选取 / 关键词搜索）和一套加权评分系统；MCP 工具名称精确匹配得 12 分，描述词边界匹配得 2 分，`+` 前缀将关键词标记为必选条件。
- 延迟加载有三种模式（tst/tst-auto/standard），`tst-auto` 在 MCP 工具描述超过上下文窗口 10% 时自动触发，默认行为是始终延迟（tst）。
- `tool_reference` 是 API 层面的一等公民，不是让模型解析的 JSON 字符串 -- API 服务端将引用展开为完整工具 schema 注入模型上下文。
- Compaction 通过 `preCompactDiscoveredTools` 快照保护已发现的工具名，避免压缩消息时丢失 `tool_reference` 导致工具失联。
- MCP 服务器连接分批建立（本地 3 并发 / 远程 20 并发），失败后指数退避重连（1s-30s，最多 5 次）。
