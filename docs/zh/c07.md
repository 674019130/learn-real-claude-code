# 记忆系统：文件型持久化与 Auto-Dream

> MEMORY.md 是索引不是数据库——200 行上限，超出则截断并警告模型。

## 概述

每个 LLM 会话本质上是健忘的。Claude Code 通过三层机制解决跨会话持久化：(1) `MEMORY.md` 索引文件，每次会话自动加载；(2) topic 文件（独立 .md），按需通过 AI 选择器召回；(3) Auto-Dream 后台整合，定期将零散信息合并为结构化记忆。

源码中没有"Dream 四阶段子 agent"这样的抽象——真实实现是一个 forked agent 执行一段精心设计的 prompt，用锁文件做并发控制，用 `mtime` 做时间门控。

## 记忆目录结构

记忆存储在 `~/.claude/projects/<sanitized-path>/memory/` 下。路径计算逻辑在 `memdir/paths.ts` 中：

```typescript
// memdir/paths.ts:223
export const getAutoMemPath = memoize(
  (): string => {
    const override = getAutoMemPathOverride() ?? getAutoMemPathSetting()
    if (override) {
      return override
    }
    const projectsDir = join(getMemoryBaseDir(), 'projects')
    return (
      join(projectsDir, sanitizePath(getAutoMemBase()), AUTO_MEM_DIRNAME) + sep
    ).normalize('NFC')
  },
  () => getProjectRoot(),
)
```

`getAutoMemBase()` 使用 git 仓库的 canonical root（而不是 cwd），所以同一个仓库的所有 worktree 共享一份记忆。结果被 `memoize` 缓存，避免每次渲染都 stat 文件系统。

## MEMORY.md：有硬上限的索引

MEMORY.md 不是存储所有记忆的地方——它是一个**索引**，每行指向一个 topic 文件。有严格的双重上限：

```typescript
// memdir/memdir.ts:37
export const ENTRYPOINT_NAME = 'MEMORY.md'
export const MAX_ENTRYPOINT_LINES = 200
export const MAX_ENTRYPOINT_BYTES = 25_000
```

截断逻辑在 `truncateEntrypointContent()` 中，先按行截断，再按字节截断：

```typescript
// memdir/memdir.ts:57
export function truncateEntrypointContent(raw: string): EntrypointTruncation {
  const trimmed = raw.trim()
  const contentLines = trimmed.split('\n')
  const lineCount = contentLines.length
  const byteCount = trimmed.length

  const wasLineTruncated = lineCount > MAX_ENTRYPOINT_LINES
  const wasByteTruncated = byteCount > MAX_ENTRYPOINT_BYTES

  if (!wasLineTruncated && !wasByteTruncated) {
    return { content: trimmed, lineCount, byteCount, wasLineTruncated, wasByteTruncated }
  }

  let truncated = wasLineTruncated
    ? contentLines.slice(0, MAX_ENTRYPOINT_LINES).join('\n')
    : trimmed

  if (truncated.length > MAX_ENTRYPOINT_BYTES) {
    const cutAt = truncated.lastIndexOf('\n', MAX_ENTRYPOINT_BYTES)
    truncated = truncated.slice(0, cutAt > 0 ? cutAt : MAX_ENTRYPOINT_BYTES)
  }

  // 截断时追加警告，模型可见
  return {
    content: truncated +
      `\n\n> WARNING: ${ENTRYPOINT_NAME} is ${reason}. Only part of it was loaded...`,
    // ...
  }
}
```

字节上限（25KB）是为了防止"少于 200 行但每行巨长"的情况——p97 分位数已观察到单个 MEMORY.md 在 200 行以内达到 197KB。

## 四类记忆类型

记忆文件使用 frontmatter 标记类型：

```typescript
// memdir/memoryTypes.ts:14
export const MEMORY_TYPES = [
  'user',       // 用户角色、偏好、背景
  'feedback',   // 工作方式反馈（纠正和确认）
  'project',    // 项目上下文（非代码可推导的）
  'reference',  // 外部系统指针
] as const
```

prompt 中明确排除了代码可推导的信息：

```typescript
// memdir/memoryTypes.ts:183
export const WHAT_NOT_TO_SAVE_SECTION: readonly string[] = [
  '## What NOT to save in memory',
  '',
  '- Code patterns, conventions, architecture, file paths, or project structure',
  '- Git history, recent changes, or who-changed-what',
  '- Debugging solutions or fix recipes',
  '- Anything already documented in CLAUDE.md files.',
  '- Ephemeral task details: in-progress work, temporary state',
]
```

### 为什么不用向量数据库

记忆系统完全基于文件，没有 embedding 或向量检索。原因：(1) 依赖外部数据库会增加运行时复杂度和失败点；(2) 200 个以下的 .md 文件用 AI 选择器已经足够精确；(3) 文件系统天然支持 git 版本控制和人工编辑。

## findRelevantMemories：AI 驱动的按需召回

当会话开始或用户提问时，系统不加载所有记忆文件——它用一个 Sonnet side-query 选择最多 5 个相关文件：

```typescript
// memdir/findRelevantMemories.ts:39
export async function findRelevantMemories(
  query: string,
  memoryDir: string,
  signal: AbortSignal,
  recentTools: readonly string[] = [],
  alreadySurfaced: ReadonlySet<string> = new Set(),
): Promise<RelevantMemory[]> {
  const memories = (await scanMemoryFiles(memoryDir, signal)).filter(
    m => !alreadySurfaced.has(m.filePath),
  )
  if (memories.length === 0) return []

  const selectedFilenames = await selectRelevantMemories(
    query, memories, signal, recentTools,
  )
  // ...
}
```

选择器使用 Sonnet 模型，接收记忆文件的 manifest（文件名 + frontmatter description），返回最多 5 个文件名：

```typescript
// memdir/findRelevantMemories.ts:18
const SELECT_MEMORIES_SYSTEM_PROMPT = `You are selecting memories that will
be useful to Claude Code as it processes a user's query. You will be given
the user's query and a list of available memory files with their filenames
and descriptions.

Return a list of filenames for the memories that will clearly be useful
(up to 5). Only include memories that you are certain will be helpful...`
```

`recentTools` 参数是一个精巧的去噪机制：如果模型正在用某个工具（如 `mcp__X__spawn`），选择器不会再召回该工具的参考文档——模型已经在用它了，不需要文档。但 gotchas/known issues 类记忆仍然会被选中。

扫描阶段（`memoryScan.ts`）只读每个 .md 文件的前 30 行 frontmatter（`FRONTMATTER_MAX_LINES = 30`），不读正文。结果按 mtime 降序排列，取前 200 个（`MAX_MEMORY_FILES = 200`）。`Promise.allSettled` 确保单个文件读取失败不会阻塞整个扫描。

## Auto-Dream：基于锁文件的后台整合

Auto-Dream 不是用户手动触发的"做梦"——它是一个后台任务，在满足条件时自动运行。

### 三重门控

门控按成本从低到高排列（`autoDream.ts:8` 注释）：(1) 时间：距上次整合 >= 24 小时（一个 stat 调用）；(2) 会话数：上次整合后有 >= 5 个新会话；(3) 锁：没有其他进程正在整合。默认阈值 `minHours: 24, minSessions: 5`（可通过 GrowthBook 远程调整）。

时间门控只需要一个 stat 调用——读取锁文件的 mtime：

```typescript
// services/autoDream/consolidationLock.ts:29
export async function readLastConsolidatedAt(): Promise<number> {
  try {
    const s = await stat(lockPath())
    return s.mtimeMs
  } catch {
    return 0  // 文件不存在 = 从未整合
  }
}
```

锁文件的 mtime 就是 `lastConsolidatedAt`——不需要额外的状态文件。文件内容是持有者的 PID，用于检测崩溃后的过期锁。

### 为什么不用数据库时间戳

用 mtime 而不是数据库记录有几个优势：(1) 零依赖，不需要额外进程；(2) 天然支持跨进程协调（文件系统是共享的）；(3) 锁获取和时间戳更新是一个原子操作（write 自动更新 mtime）。

### 整合 prompt：四阶段指令

`consolidationPrompt.ts` 构建的 prompt 包含四个阶段：Phase 1 Orient（ls 记忆目录、读 MEMORY.md 和现有 topic 文件）、Phase 2 Gather（从日志和 transcript 中搜集新信号）、Phase 3 Consolidate（合并到现有 topic 文件、转换相对日期、删除矛盾事实）、Phase 4 Prune（更新 MEMORY.md 索引，保持在 200 行 / 25KB 以内）。

关键约束：Dream agent 的 Bash 被限制为只读命令：

```typescript
// services/autoDream/autoDream.ts:216
const extra = `
**Tool constraints for this run:** Bash is restricted to read-only commands
(ls, find, grep, cat, stat, wc, head, tail, and similar). Anything that
writes, redirects to a file, or modifies state will be denied.`
```

这个约束不是靠 prompt 实现的——它通过 `createAutoMemCanUseTool` 在代码层面强制：

```typescript
// services/extractMemories/extractMemories.ts:171
export function createAutoMemCanUseTool(memoryDir: string): CanUseToolFn {
  return async (tool, input) => {
    // Read/Grep/Glob 无限制
    if (tool.name === FILE_READ_TOOL_NAME || tool.name === GREP_TOOL_NAME
        || tool.name === GLOB_TOOL_NAME) {
      return { behavior: 'allow', updatedInput: input }
    }
    // Bash 只允许 isReadOnly 的命令
    if (tool.name === BASH_TOOL_NAME) {
      const parsed = tool.inputSchema.safeParse(input)
      if (parsed.success && tool.isReadOnly(parsed.data)) {
        return { behavior: 'allow', updatedInput: input }
      }
      return denyAutoMemTool(tool, 'only read-only bash allowed')
    }
    // Edit/Write 只允许写入 memory 目录
    if (tool.name === FILE_EDIT_TOOL_NAME || tool.name === FILE_WRITE_TOOL_NAME) {
      const filePath = (input as any).file_path
      if (typeof filePath === 'string' && isAutoMemPath(expandPath(filePath))) {
        return { behavior: 'allow', updatedInput: input }
      }
      return denyAutoMemTool(tool, 'writes only allowed in memory dir')
    }
  }
}
```

这就是真正的"只读 bash 子 agent"——不是一个独立的 agent 类型，而是一个 `canUseTool` 回调函数，对每次工具调用做精细化权限控制。

### Extract Memories：与 Dream 互斥

除了定期的 Auto-Dream，每次 query 结束后还有一个 `extractMemories` 进程，从当前会话提取值得记住的信息。它与 Dream 是互斥的：

```typescript
// services/extractMemories/extractMemories.ts:121
function hasMemoryWritesSince(
  messages: Message[],
  sinceUuid: string | undefined,
): boolean {
  // 如果主 agent 已经写了记忆文件，跳过提取
  for (const block of content) {
    const filePath = getWrittenFilePath(block)
    if (filePath !== undefined && isAutoMemPath(filePath)) {
      return true
    }
  }
  return false
}
```

如果主 agent 在对话中已经手动写了记忆（用户说"记住这个"），后台提取就跳过——避免重复写入。

## 验证

以下实验可以在源码中验证本章描述的机制：

1. **验证 MEMORY.md 上限**：在 `memdir/memdir.ts` 中确认 `MAX_ENTRYPOINT_LINES = 200` 和 `MAX_ENTRYPOINT_BYTES = 25_000`。然后看 `truncateEntrypointContent` 如何在截断后追加 `> WARNING:` 消息。

2. **验证记忆类型**：在 `memdir/memoryTypes.ts` 中确认 `MEMORY_TYPES = ['user', 'feedback', 'project', 'reference']`。在 `WHAT_NOT_TO_SAVE_SECTION` 中确认排除了代码模式和 git 历史。

3. **验证 Dream 门控阈值**：在 `services/autoDream/autoDream.ts` 中确认 `DEFAULTS = { minHours: 24, minSessions: 5 }`。在 `consolidationLock.ts` 中确认锁文件的 mtime 就是 lastConsolidatedAt。

4. **验证只读约束**：在 `services/extractMemories/extractMemories.ts` 中找到 `createAutoMemCanUseTool`，确认 Bash 只允许 `isReadOnly` 的命令，Edit/Write 只允许写入 memory 目录。

5. **查看你自己的记忆目录**：运行 `ls ~/.claude/projects/` 看到按项目路径组织的记忆目录，每个目录下有 `MEMORY.md` 和若干 topic 文件。

## 关键收获

- MEMORY.md 是索引而非数据库，有 200 行 + 25KB 双重上限；截断时追加 WARNING 消息让模型知道索引不完整。
- 记忆类型限定为四种（user/feedback/project/reference），代码可推导的信息被明确排除——grep 和 git log 比记忆文件更可靠。
- `findRelevantMemories` 用 Sonnet side-query 从 frontmatter manifest 中选择最多 5 个相关文件，不读正文、不用向量数据库。
- Auto-Dream 用三重门控（时间 > 会话数 > 锁文件）控制触发频率，锁文件的 mtime 兼作 lastConsolidatedAt 时间戳。
- Dream agent 的"只读 Bash"不是靠 prompt 指令实现的，而是通过 `createAutoMemCanUseTool` 回调在代码层面强制——Bash 只允许 `isReadOnly` 的命令，Write/Edit 只允许写入 memory 目录。
