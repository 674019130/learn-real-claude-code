# 第十四章 -- 可观测性：OTel 指标、逐模型成本追踪与错误恢复管道

> `withRetry` 是 AsyncGenerator——每次重试等待时 yield 出 `SystemAPIErrorMessage`，让 UI 实时显示倒计时而不阻塞控制流。

---

## 概述

生产级 agent 在三个维度上需要工程保障：知道发生了什么（遥测）、知道花了多少（成本追踪）、以及在失败后自动恢复（重试管道）。Claude Code 的实现远比"出错就重试"复杂：

1. **OTel 指标**：通过 OpenTelemetry `Meter` 上报 session/cost/token/LOC 等计数器。
2. **逐模型成本追踪**：`cost-tracker.ts` 按模型累积 input/output/cache-read/cache-write token，实时计算美元成本。
3. **重试管道**：`withRetry` 是 AsyncGenerator，区分 529（过载）和 429（限流），支持 fast mode 降级、模型 fallback、无人值守无限重试。

本章所有代码块均来自真实源码，标注文件名和行号。

### 为什么不用 StatsD？

旧版 Claude Code 用 Datadog StatsD。OpenTelemetry 是厂商无关标准——企业用户用 Bedrock/Vertex 时，OTEL 数据可以路由到他们自己的 Grafana/Datadog/New Relic。StatsD 是 Datadog 专属协议，锁定一个后端。

## 2. 逐模型成本追踪

`cost-tracker.ts` 不只是一个总计数器——它按模型名聚合所有 token 类别：

```typescript
// cost-tracker.ts:71-80
type StoredCostState = {
  totalCostUSD: number
  totalAPIDuration: number
  totalAPIDurationWithoutRetries: number
  totalToolDuration: number
  totalLinesAdded: number
  totalLinesRemoved: number
  lastDuration: number | undefined
  modelUsage: { [modelName: string]: ModelUsage } | undefined
}
```

每次 API 调用完成后，`addToTotalSessionCost` 累积到对应模型：

```typescript
// cost-tracker.ts:250-276
function addToTotalModelUsage(
  cost: number,
  usage: Usage,
  model: string,
): ModelUsage {
  const modelUsage = getUsageForModel(model) ?? {
    inputTokens: 0,
    outputTokens: 0,
    cacheReadInputTokens: 0,
    cacheCreationInputTokens: 0,
    webSearchRequests: 0,
    costUSD: 0,
    contextWindow: 0,
    maxOutputTokens: 0,
  }

  modelUsage.inputTokens += usage.input_tokens
  modelUsage.outputTokens += usage.output_tokens
  modelUsage.cacheReadInputTokens += usage.cache_read_input_tokens ?? 0
  modelUsage.cacheCreationInputTokens += usage.cache_creation_input_tokens ?? 0
  modelUsage.webSearchRequests +=
    usage.server_tool_use?.web_search_requests ?? 0
  modelUsage.costUSD += cost
  modelUsage.contextWindow = getContextWindowForModel(model, getSdkBetas())
  modelUsage.maxOutputTokens = getModelMaxOutputTokens(model).default
  return modelUsage
}
```

注意 `cacheReadInputTokens` 和 `cacheCreationInputTokens` 的分别追踪。Cache read 的单价约为普通 input 的 10%，精确追踪这两个数字才能准确计算 prompt cache 带来的成本节省。

成本同时上报到 OTel 计数器，区分 fast mode：

```typescript
// cost-tracker.ts:287-300
const attrs =

```

advisor（嵌套模型调用）的成本递归累积——一次主调用可能触发多个 advisor 子调用：

```typescript
// cost-tracker.ts:302-321
for (const advisorUsage of getAdvisorUsage(usage)) {
  const advisorCost = calculateUSDCost(advisorUsage.model, advisorUsage)
  logEvent('tengu_advisor_tool_token_usage', {
    advisor_model: advisorUsage.model as ...,
    // ...
    cost_usd_micros: Math.round(advisorCost * 1_000_000),
  })
  totalCost += addToTotalSessionCost(
    advisorCost,
    advisorUsage,
    advisorUsage.model,
  )
}
```

会话成本持久化到项目配置，支持 `--resume` 恢复累积成本：

```typescript
// cost-tracker.ts:130-137
export function restoreCostStateForSession(sessionId: string): boolean {
  const data = getStoredSessionCosts(sessionId)
  if (!data) {
    return false
  }
  setCostStateForRestore(data)
  return true
}
```

### 为什么不用 per-request 日志替代内存聚合？

每次 API 调用写日志然后汇总，在 100+ 次调用的长会话中产生大量磁盘 I/O。内存聚合只在会话结束或 `saveCurrentSessionCosts` 时写一次。终端右上角的实时成本显示读内存变量，不做任何 I/O。

## 3. withRetry：AsyncGenerator 重试管道

`withRetry` 不是普通的 async 函数——它是 `AsyncGenerator`，每次重试等待时 yield 出 `SystemAPIErrorMessage`，让 UI 显示实时倒计时：

```typescript
// services/api/withRetry.ts:170-178
export async function* withRetry<T>(
  getClient: () => Promise<Anthropic>,
  operation: (
    client: Anthropic,
    attempt: number,
    context: RetryContext,
  ) => Promise<T>,
  options: RetryOptions,
): AsyncGenerator<SystemAPIErrorMessage, T> {
```

核心参数：最多 10 次重试（`DEFAULT_MAX_RETRIES = 10`），基础延迟 500ms 指数增长：

```typescript
// services/api/withRetry.ts:54-55
const DEFAULT_MAX_RETRIES = 10
export const BASE_DELAY_MS = 500
```

指数退避加随机 jitter（25% 范围）防止雷群效应：

```typescript
// services/api/withRetry.ts:530-548
export function getRetryDelay(
  attempt: number,
  retryAfterHeader?: string | null,
  maxDelayMs = 32000,
): number {
  if (retryAfterHeader) {
    const seconds = parseInt(retryAfterHeader, 10)
    if (!isNaN(seconds)) {
      return seconds * 1000
    }
  }

  const baseDelay = Math.min(
    BASE_DELAY_MS * Math.pow(2, attempt - 1),
    maxDelayMs,
  )
  const jitter = Math.random() * 0.25 * baseDelay
  return baseDelay + jitter
}
```

## 4. 529 错误处理：三次连续失败触发 fallback

529（Overloaded）不是普通的重试场景。连续 3 次 529 后触发模型 fallback：

```typescript
// services/api/withRetry.ts:53
const MAX_529_RETRIES = 3
```

```typescript
// services/api/withRetry.ts:327-351
if (is529Error(error) && ...) {
  consecutive529Errors++
  if (consecutive529Errors >= MAX_529_RETRIES) {
    if (options.fallbackModel) {
      logEvent('tengu_api_opus_fallback_triggered', {
        original_model: options.model as ...,
        fallback_model: options.fallbackModel as ...,
        provider: getAPIProviderForStatsig(),
      })
      throw new FallbackTriggeredError(
        options.model,
        options.fallbackModel,
      )
    }
    // ...
  }
}
```

非前台来源的 529 直接放弃，不重试——防止后台任务（摘要、标题生成、分类器）在容量级联时放大流量：

```typescript
// services/api/withRetry.ts:62-82
const FOREGROUND_529_RETRY_SOURCES = new Set<QuerySource>([
  'repl_main_thread',
  // ...
  'auto_mode',
  ...(feature('BASH_CLASSIFIER') ? (['bash_classifier'] as const) : []),
])
```

```typescript
// services/api/withRetry.ts:318-324
if (is529Error(error) && !shouldRetry529(options.querySource)) {
  logEvent('tengu_api_529_background_dropped', {
    query_source: options.querySource as ...,
  })
  throw new CannotRetryError(error, retryContext)
}
```

## 5. Fast mode 降级

Fast mode 遇到 429/529 时的降级策略是分层的：

```typescript
// services/api/withRetry.ts:284-305
const retryAfterMs = getRetryAfterMs(error)
if (retryAfterMs !== null && retryAfterMs < SHORT_RETRY_THRESHOLD_MS) {
  // Short retry-after (<20s): wait and retry with fast mode still active
  await sleep(retryAfterMs, options.signal, { abortError })
  continue
}
// Long or unknown retry-after: enter cooldown (switches to standard speed)
const cooldownMs = Math.max(
  retryAfterMs ?? DEFAULT_FAST_MODE_FALLBACK_HOLD_MS,
  MIN_COOLDOWN_MS,
)
triggerFastModeCooldown(Date.now() + cooldownMs, cooldownReason)
retryContext.fastMode = false
continue
```

- **短延迟 (<20s)**：保持 fast mode，等待后重试——保留 prompt cache（模型名不变）。
- **长延迟 (>=20s)**：进入 cooldown，降级到标准模型，至少 10 分钟不尝试 fast mode。

```typescript
// services/api/withRetry.ts:799-801
const DEFAULT_FAST_MODE_FALLBACK_HOLD_MS = 30 * 60 * 1000 // 30 minutes
const SHORT_RETRY_THRESHOLD_MS = 20 * 1000 // 20 seconds
const MIN_COOLDOWN_MS = 10 * 60 * 1000 // 10 minutes
```

### 为什么不直接用 circuit breaker 模式？

经典 circuit breaker 是二元的（OPEN/CLOSED）。Claude Code 的重试管道是多层渐进的：先在同模型重试 -> 降级模型 -> fallback 模型 -> 放弃。529 不等于"服务挂了"——可能只是当前模型过载，换个模型就好。circuit breaker 的"完全断开"对 agent 体验太粗暴。

## 6. 无人值守无限重试

ant 专属的 `CLAUDE_CODE_UNATTENDED_RETRY` 模式在 429/529 时无限重试，支持长达 6 小时的等待窗口：

```typescript
// services/api/withRetry.ts:96-98
const PERSISTENT_MAX_BACKOFF_MS = 5 * 60 * 1000
const PERSISTENT_RESET_CAP_MS = 6 * 60 * 1000 * 1000
const HEARTBEAT_INTERVAL_MS = 30_000
```

长等待被分块为 30 秒心跳，每次 yield 出 `SystemAPIErrorMessage`。这防止宿主环境（Kubernetes、CI runner）因长时间无 stdout 而标记进程为 idle：

```typescript
// services/api/withRetry.ts:489-501
let remaining = delayMs
while (remaining > 0) {
  if (options.signal?.aborted) throw new APIUserAbortError()
  if (error instanceof APIError) {
    yield createSystemAPIErrorMessage(
      error,
      remaining,
      reportedAttempt,
      maxRetries,
    )
  }
  const chunk = Math.min(remaining, HEARTBEAT_INTERVAL_MS)
  await sleep(chunk, options.signal, { abortError })
  remaining -= chunk
}
```

`attempt` 被钳位在 `maxRetries` 防止 for 循环终止，真正的尝试次数用 `persistentAttempt` 跟踪：

```typescript
// services/api/withRetry.ts:506
if (attempt >= maxRetries) attempt = maxRetries
```

## 验证

1. **检查 OTel 计数器**：在 `bootstrap/state.ts:89-99` 确认 8 个 `AttributedCounter` 字段。搜索 `getCostCounter()?.add` 查看所有成本上报点。

2. **追踪成本流**：从 `cost-tracker.ts:278` 的 `addToTotalSessionCost` 开始，跟随 `addToTotalCostState` 和 `getCostCounter()?.add`，验证成本同时写入内存状态和 OTel 计数器。

3. **验证指数退避**：`getRetryDelay(1)` 应返回 ~500ms，`getRetryDelay(10)` 应返回 ~32000ms（命中 maxDelayMs 上限）。jitter 范围是 `0 ~ 0.25 * baseDelay`。

4. **触发 529 fallback**：在测试环境中用 `/mock-limits` 模拟连续 529 错误，观察第 3 次失败后 `tengu_api_opus_fallback_triggered` 事件触发和模型切换。

5. **检查 fast mode 降级阈值**：`SHORT_RETRY_THRESHOLD_MS = 20000`。`retry-after: 15` 保持 fast mode 重试；`retry-after: 30` 触发 10 分钟 cooldown 降级。

## 关键收获

- `withRetry` 是 AsyncGenerator 而非普通 async 函数：每次重试等待时 yield 出 `SystemAPIErrorMessage`，UI 实时显示倒计时而不阻塞控制流。
- 逐模型成本追踪精确区分 `input`/`output`/`cacheRead`/`cacheCreation`/`webSearch` 五种 token 类别，advisor 子调用的成本递归累积到总额。
- 529（过载）错误有专门的 `consecutive529Errors` 计数器，3 次连续失败触发 `FallbackTriggeredError` 切换模型；非前台来源的 529 直接放弃不重试，防止容量级联放大。
- fast mode 降级是分层的：短延迟 (<20s) 保持 fast mode 重试以保留 prompt cache；长延迟触发至少 10 分钟的 cooldown 切换到标准模型。
- 无人值守模式将长等待分块为 30 秒心跳 yield，防止宿主环境标记进程 idle；`attempt` 钳位防止 for 循环终止，`persistentAttempt` 跟踪真实尝试次数。
