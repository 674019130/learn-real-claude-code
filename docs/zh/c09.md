# Hook 生命周期：工具执行的切面

> 27 种 hook 事件、4 种 hook 后端（command/http/prompt/agent）、异步注册表 -- 真实的切面系统远比文档暗示的复杂。

## 概述

Hook 系统是 Claude Code 最强大的扩展点。它在工具执行的每个关键节点插入自定义逻辑，覆盖从 session 启动到文件变更的完整生命周期。

但真实源码与"三种 hook 类型"的简化描述差距巨大：实际有 **27 种 hook 事件**、**4 种持久化后端** 加上一种运行时 function hook、一个完整的**异步 hook 注册表**，以及一套精密的 JSON 输出验证管线。

## 4 种持久化后端 + 1 种运行时后端

源码中 hook 类型是一个 discriminated union，比文档描述多一种 `agent` 后端：

```typescript

    BashCommandHookSchema,
```

第 5 种是运行时 function hook，无法持久化到 settings.json，只存在于内存中：

## Hook 输出的 JSON 验证管线

Hook 不是简单的 pass/fail。源码中有一套完整的 Zod schema 验证每个 hook 的 JSON 输出，支持 per-event 的 specific output：

```typescript
// types/hooks.ts:50
export const syncHookResponseSchema = lazySchema(() =>
  z.object({
    continue: z.boolean().optional(),
    suppressOutput: z.boolean().optional(),
    stopReason: z.string().optional(),
    decision: z.enum(['approve', 'block']).optional(),
    reason: z.string().optional(),
    hookSpecificOutput: z
      .union([
        z.object({
          hookEventName: z.literal('PreToolUse'),
          permissionDecision: permissionBehaviorSchema().optional(),
          updatedInput: z.record(z.string(), z.unknown()).optional(),
          additionalContext: z.string().optional(),
        }),
        z.object({
          hookEventName: z.literal('PostToolUse'),
          additionalContext: z.string().optional(),
          updatedMCPToolOutput: z.unknown().optional(),
        }),
        // ... 12 种 event-specific schemas
      ])
      .optional(),
  }),
)
```

关键细节：`PreToolUse` 的 `hookSpecificOutput` 可以同时返回 `permissionDecision`（allow/deny/ask）和 `updatedInput`（修改工具输入），这意味着一个 hook 可以在批准执行的同时修改参数。

## 异步 Hook 注册表

Hook 可以返回 `{ async: true }` 声明自己是异步的，Claude Code 会在后台持续轮询结果：

```typescript
// utils/hooks/AsyncHookRegistry.ts:28
const pendingHooks = new Map<string, PendingAsyncHook>()

// utils/hooks/AsyncHookRegistry.ts:30
export function registerPendingAsyncHook({
  processId, hookId, asyncResponse, hookName, hookEvent,
  command, shellCommand, toolName, pluginId,
}: { ... }): void {
  const timeout = asyncResponse.asyncTimeout || 15000 // Default 15s
  const stopProgressInterval = startHookProgressInterval({
    hookId, hookName, hookEvent,
    getOutput: async () => { ... },
  })
  pendingHooks.set(processId, {
    processId, hookId, hookName, hookEvent,
    startTime: Date.now(), timeout,
    responseAttachmentSent: false, shellCommand, stopProgressInterval,
  })
}
```

轮询逻辑在 `checkForAsyncHookResponses` 中 -- 它遍历所有 pending hooks，检查 shell command 是否完成，解析 stdout 中的 JSON 行：

```typescript
// utils/hooks/AsyncHookRegistry.ts:193
const lines = stdout.split('\n')
for (const line of lines) {
  if (line.trim().startsWith('{')) {
    try {
      const parsed = jsonParse(line.trim())
      if (!('async' in parsed)) {
        response = parsed
        break
      }
    } catch { /* ... */ }
  }
}
```

这意味着异步 hook 的输出协议是：先输出 `{"async":true}` 启动后台模式，完成后输出一个普通 JSON 对象作为最终结果。

## PreToolUse 的权限决策链

PreToolUse hook 返回的 `permissionDecision` 不是最终裁决。源码中有一个精密的决策链：hook 的 allow 不能绕过 settings.json 的 deny/ask 规则：

```typescript
// services/tools/toolHooks.ts:347
if (hookPermissionResult?.behavior === 'allow') {
  const hookInput = hookPermissionResult.updatedInput ?? input

  // Hook allow skips the interactive prompt, but deny/ask rules still apply.
  const ruleCheck = await checkRuleBasedPermissions(tool, hookInput, toolUseContext)
  if (ruleCheck === null) {
    return { decision: hookPermissionResult, input: hookInput }
  }
  if (ruleCheck.behavior === 'deny') {
    // deny rule overrides hook approval
    return { decision: ruleCheck, input: hookInput }
  }
  // ask rule -- dialog required despite hook approval
  return {
    decision: await canUseTool(tool, hookInput, toolUseContext, assistantMessage, toolUseID),
    input: hookInput,
  }
}
```

这是一个安全不变量：hook 可以免除交互式提示，但不能覆盖管理员设定的 deny 规则。

## 事件系统：缓冲 + 按需发射

Hook 事件不是无条件发射的。源码中有一个缓冲机制和白名单：

```typescript
// utils/hooks/hookEvents.ts:18
const ALWAYS_EMITTED_HOOK_EVENTS = ['SessionStart', 'Setup'] as const

const MAX_PENDING_EVENTS = 100

// utils/hooks/hookEvents.ts:72
function emit(event: HookExecutionEvent): void {
  if (eventHandler) {
    eventHandler(event)
  } else {
    pendingEvents.push(event)
    if (pendingEvents.length > MAX_PENDING_EVENTS) {
      pendingEvents.shift()  // Ring buffer -- 丢弃最老的
    }
  }
}
```

在 handler 注册之前，事件被缓冲到一个 100 条上限的环形队列中。只有 `SessionStart` 和 `Setup` 默认发射，其他事件需要 `includeHookEvents` 选项或 `CLAUDE_CODE_REMOTE` 模式才会发射。

## Session Hooks：Map 而非 Record

Session hooks 的存储用 Map 而不是 Record，这不是随意选择：

```typescript
// utils/hooks/sessionHooks.ts:48
/**
 * Map (not Record) so .set/.delete don't change the container's identity.
 * Mutator functions mutate the Map and return prev unchanged, letting
 * store.ts's Object.is(next, prev) check short-circuit and skip listener
 * notification. ...
 * This matters under high-concurrency workflows: parallel() with N
 * schema-mode agents fires N addFunctionHook calls in one synchronous
 * tick. With a Record + spread, each call cost O(N) to copy the growing
 * map (O(N^2) total) plus fired all ~30 store listeners.
 * With Map: .set() is O(1), return prev means zero listener fires.
 */
export type SessionHooksState = Map<string, SessionStore>
```

N 个并发 agent 同时注册 hook 时，Record 方案的时间复杂度是 O(N^2)，Map 是 O(N)。

## 为什么不用 EventEmitter？

Node.js 内置 EventEmitter 看似天然适合 hook 系统，但源码选择了自定义的事件管线。原因在 `hookEvents.ts` 中明确可见：

1. **缓冲需求**：EventEmitter 在 handler 注册前丢弃事件，但 Claude Code 需要缓冲 session 启动前的事件。
2. **选择性发射**：`shouldEmit()` 函数根据事件类型和配置决定是否发射 -- EventEmitter 没有这种过滤层。
3. **环形队列溢出**：`MAX_PENDING_EVENTS = 100` 防止内存泄漏，EventEmitter 的 listener 队列没有容量控制。

## 为什么不用 Middleware 模式？

Express/Koa 的 middleware 链是处理请求的经典模式，但 hook 系统没有采用它：

1. **异步生命周期**：Hook 可以返回 `{async:true}` 在后台运行，middleware 的 `next()` 语义无法表达"我现在返回但稍后还有结果"。
2. **多源聚合**：Hook 来自 settings.json、session 运行时、plugin 三个源头，需要在执行前 merge -- middleware 链假设单一注册源。
3. **权限叠加**：PreToolUse 的结果需要与 settings.json 的 deny/ask 规则叠加，不是简单的 pass/block -- 这超出了 middleware 的 req/res 模型。

### 实验 3：验证 hook 权限叠加行为

在 `~/.claude/settings.json` 中配置：

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "echo '{\"hookSpecificOutput\":{\"hookEventName\":\"PreToolUse\",\"permissionDecision\":\"allow\"}}'"

        ]
      }
    ]

```

## 关键收获

- Claude Code 实际有 **27 种 hook 事件**，覆盖从 `SessionStart` 到 `FileChanged` 的完整生命周期，远超文档中的 3 种。
- **4 种持久化后端**（command/http/prompt/agent）加上 1 种运行时 function hook；agent hook 会启动独立 Claude 实例做 agentic verification。
- 异步 hook 通过 `AsyncHookRegistry` 管理，使用"先输出 `{async:true}`，完成后输出结果 JSON"的两阶段协议。
- PreToolUse hook 的 `allow` 不是最终裁决 -- 源码中 `resolveHookPermissionDecision` 确保 settings.json 的 deny 规则始终优先于 hook approval。
- 事件系统使用 100 条上限的环形缓冲区，只有 `SessionStart` 和 `Setup` 默认发射，其他事件需要显式启用。
- Session hooks 用 Map 而非 Record 存储，避免 N 个并发 agent 注册时的 O(N^2) 复制开销和不必要的 listener 通知。
